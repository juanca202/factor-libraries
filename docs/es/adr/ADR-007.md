# ADR-007: Estrategia de Testing

**📖 [English](../../en/adr/ADR-007.md) | [Español](./ADR-007.md)**

**Estado:** Aceptado  
**Fecha:** 2025  
**Decisores:** Equipo de Arquitectura

## Contexto

A medida que la aplicación crece, mantener la calidad del código y prevenir regresiones se vuelve crítico. Sin una estrategia de testing bien definida:

- Los tests pueden ser inconsistentes en estructura y calidad
- Bugs críticos pueden pasar desapercibidos
- La refactorización se vuelve riesgosa sin cobertura de tests
- El onboarding de nuevos desarrolladores es más lento sin ejemplos de tests
- Los problemas de integración se descubren tarde en el desarrollo
- No hay guías claras sobre qué testear y cómo

Necesitamos una estrategia de testing exhaustiva que:
- Asegure confiabilidad y mantenibilidad del código
- Proporcione confianza al refactorizar
- Sirva como documentación del comportamiento del código
- Detecte bugs temprano en el ciclo de desarrollo
- Mantenga calidad de tests consistente en todo el equipo

## Decisión

Usaremos un **enfoque de testing multi-capa** con:

1. **Unit Testing:** Vitest para testear unidades individuales (componentes, servicios, utilidades) en aislamiento
2. **Integration Testing:** Testing de interacciones componente-servicio
3. **E2E Testing:** Playwright para flujos de trabajo de usuario end-to-end
4. **Patrón AAA:** Estructura Arrange, Act, Assert para todos los tests
5. **Cobertura de Tests:** Objetivo de ≥80% de cobertura de ramas, enfocándose en rutas críticas
6. **Aislamiento de Tests:** Los tests deben ser independientes y determinísticos

## Implementación

### Stack de Testing

- **Vitest:** Framework de testing unitario e integración
- **@analogjs/vite-plugin-angular:** Configuración específica de Vitest para Angular
- **Playwright:** Framework de testing end-to-end
- **Utilidades de Testing:** Utilidades de testing de Angular y harnesses de Material

### Unit Testing con Vitest

#### Estructura de Archivo de Test

```typescript
describe('ComponentName', () => {
  // Arrange
  let component: ComponentName;
  let fixture: ComponentFixture<ComponentName>;

  beforeEach(() => {
    // Setup
  });

  describe('MethodName', () => {
    it('should do something when condition is met', () => {
      // Arrange
      const input = 'test value';
      
      // Act
      const result = component.methodName(input);
      
      // Assert
      expect(result).toBe('expected value');
    });

    it('should handle error case', () => {
      // Arrange
      const invalidInput = null;
      
      // Act & Assert
      expect(() => component.methodName(invalidInput)).toThrow();
    });
  });
});
```

#### Patrón AAA (Arrange, Act, Assert)

Todos los tests deben seguir el patrón AAA:

```typescript
it('should calculate total price correctly', () => {
  // Arrange: Configurar datos de test y condiciones
  const items = [
    { price: 10, quantity: 2 },
    { price: 5, quantity: 3 }
  ];
  component.items = items;

  // Act: Ejecutar el código bajo test
  const total = component.calculateTotal();

  // Assert: Verificar el resultado esperado
  expect(total).toBe(35);
});
```

#### Testing de Componentes

```typescript
import { ComponentFixture, TestBed } from '@angular/core/testing';
import { CustomerListComponent } from './customer-list.component';
import { CustomerRepository } from '@/features/customers/repositories/customer-repository';
import { of } from 'rxjs';

describe('CustomerListComponent', () => {
  let component: CustomerListComponent;
  let fixture: ComponentFixture<CustomerListComponent>;
  let mockRepository: Partial<CustomerRepository>;

  beforeEach(async () => {
    // Arrange: Crear mock repository
    mockRepository = {
      findBy: vi.fn().mockReturnValue({
        load: vi.fn().mockResolvedValue([
          { id: '1', name: 'John' },
          { id: '2', name: 'Jane' }
        ]),
        value: signal([]),
        loading: signal(false),
        error: signal(null),
        destroy: vi.fn()
      })
    } as any;

    await TestBed.configureTestingModule({
      imports: [CustomerListComponent],
      providers: [
        { provide: CustomerRepository, useValue: mockRepository }
      ]
    }).compileComponents();

    fixture = TestBed.createComponent(CustomerListComponent);
    component = fixture.componentInstance;
  });

  it('should load customers on init', async () => {
    // Arrange
    const loadSpy = vi.spyOn(mockRepository.findBy()!, 'load');

    // Act
    component.ngOnInit();
    await fixture.whenStable();

    // Assert
    expect(loadSpy).toHaveBeenCalled();
  });

  it('should display customers when loaded', () => {
    // Arrange
    component.customers = {
      value: signal([{ id: '1', name: 'John' }]),
      loading: signal(false),
      error: signal(null),
      load: jest.fn(),
      destroy: jest.fn()
    } as any;
    fixture.detectChanges();

    // Act
    const customerElements = fixture.nativeElement.querySelectorAll('.customer-item');

    // Assert
    expect(customerElements.length).toBe(1);
  });
});
```

#### Testing de Servicios

```typescript
import { TestBed } from '@angular/core/testing';
import { HttpClientTestingModule, HttpTestingController } from '@angular/common/http/testing';
import { CustomerRepository } from './customer-repository';

describe('CustomerRepository', () => {
  let repository: CustomerRepository;
  let httpMock: HttpTestingController;

  beforeEach(() => {
    TestBed.configureTestingModule({
      imports: [HttpClientTestingModule],
      providers: [CustomerRepository]
    });

    repository = TestBed.inject(CustomerRepository);
    httpMock = TestBed.inject(HttpTestingController);
  });

  afterEach(() => {
    httpMock.verify();
  });

  it('should fetch customer by id', async () => {
    // Arrange
    const customerId = '123';
    const mockCustomer = { id: customerId, name: 'John' };

    // Act
    const resource = repository.find();
    const promise = resource.load(customerId);

    // Assert
    const req = httpMock.expectOne(`/api/v1/customers/${customerId}`);
    expect(req.request.method).toBe('GET');
    req.flush(mockCustomer);

    const result = await promise;
    expect(result).toEqual(mockCustomer);
    expect(resource.value()).toEqual(mockCustomer);
  });
});
```

#### Testing de Signals

```typescript
describe('CustomerService', () => {
  let service: CustomerService;

  beforeEach(() => {
    service = new CustomerService();
  });

  it('should initialize with empty customers', () => {
    // Arrange & Act
    const customers = service.customers();

    // Assert
    expect(customers).toEqual([]);
  });

  it('should update customers signal', () => {
    // Arrange
    const newCustomers = [{ id: '1', name: 'John' }];

    // Act
    service.setCustomers(newCustomers);

    // Assert
    expect(service.customers()).toEqual(newCustomers);
  });

  it('should compute total count', () => {
    // Arrange
    service.setCustomers([
      { id: '1', name: 'John' },
      { id: '2', name: 'Jane' }
    ]);

    // Act
    const count = service.totalCount();

    // Assert
    expect(count).toBe(2);
  });
});
```

### Mejores Prácticas de Testing

#### 1. Nombres de Tests Descriptivos

```typescript
// ✅ Correcto - describe el comportamiento esperado
it('should display error message when API call fails', () => {});
it('should disable submit button when form is invalid', () => {});
it('should filter customers by search term', () => {});

// ❌ Incorrecto - vago o enfocado en implementación
it('should work', () => {});
it('should call method', () => {});
it('test1', () => {});
```

#### 2. Testear Casos Positivos y Negativos

```typescript
describe('validateEmail', () => {
  it('should return true for valid email', () => {
    expect(validateEmail('test@example.com')).toBe(true);
  });

  it('should return false for invalid email', () => {
    expect(validateEmail('invalid-email')).toBe(false);
  });

  it('should return false for empty string', () => {
    expect(validateEmail('')).toBe(false);
  });

  it('should return false for null', () => {
    expect(validateEmail(null as any)).toBe(false);
  });
});
```

#### 3. Usar Test Doubles (Mocks, Spies, Stubs)

```typescript
import { vi } from 'vitest';

// Mock dependencies
const mockRepository = {
  findBy: vi.fn().mockReturnValue({
    load: vi.fn(),
    value: signal([]),
    loading: signal(false),
    error: signal(null),
    destroy: vi.fn()
  })
} as any;

// Spy on methods
const loadSpy = vi.spyOn(component, 'loadData');

// Stub external services
vi.mock('app/core/services/logger', () => ({
  Logger: {
    log: vi.fn()
  }
}));
```

#### 4. Testear APIs Públicas, No Detalles de Implementación

```typescript
// ✅ Correcto - testear comportamiento público
it('should update customer name', () => {
  component.updateCustomerName('123', 'New Name');
  expect(component.getCustomer('123').name).toBe('New Name');
});

// ❌ Incorrecto - testear detalles de implementación
it('should call internal method', () => {
  const spy = jest.spyOn(component as any, 'internalMethod');
  component.updateCustomerName('123', 'New Name');
  expect(spy).toHaveBeenCalled();
});
```

#### 5. Mantener Tests Aislados

```typescript
// ✅ Correcto - cada test es independiente
describe('Calculator', () => {
  let calculator: Calculator;

  beforeEach(() => {
    calculator = new Calculator(); // Instancia fresca para cada test
  });

  it('should add numbers', () => {
    expect(calculator.add(2, 3)).toBe(5);
  });

  it('should subtract numbers', () => {
    expect(calculator.subtract(5, 3)).toBe(2);
  });
});

// ❌ Incorrecto - tests dependen unos de otros
let calculator = new Calculator(); // Estado compartido

it('should add numbers', () => {
  calculator.add(2, 3);
  expect(calculator.result).toBe(5);
});

it('should subtract from previous result', () => {
  calculator.subtract(2); // Depende del test anterior
  expect(calculator.result).toBe(3);
});
```

#### 6. Usar Angular Material Harnesses

```typescript
import { MatButtonHarness } from '@angular/material/button/testing';
import { TestbedHarnessEnvironment } from '@angular/cdk/testing/testbed';

it('should disable button when form is invalid', async () => {
  // Arrange
  component.form.setErrors({ invalid: true });
  fixture.detectChanges();

  // Act
  const loader = TestbedHarnessEnvironment.loader(fixture);
  const button = await loader.getHarness(MatButtonHarness);

  // Assert
  expect(await button.isDisabled()).toBe(true);
});
```

### Objetivos de Cobertura de Tests

- **Cobertura de Ramas:** ≥80% para rutas críticas
- **Cobertura de Funciones:** ≥80% general
- **Cobertura de Líneas:** ≥80% general
- **Áreas de Enfoque:** Lógica de negocio, manejo de errores, casos extremos

### E2E Testing con Playwright

Para testing end-to-end, usar Playwright para testear flujos de trabajo completos de usuario:

```typescript
import { test, expect } from '@playwright/test';

test('user can login and view dashboard', async ({ page }) => {
  // Arrange: Navegar a página de login
  await page.goto('/signin');

  // Act: Llenar formulario de login y enviar
  await page.fill('[formControlName="email"]', 'user@example.com');
  await page.fill('[formControlName="password"]', 'password123');
  await page.click('button[type="submit"]');

  // Assert: Verificar redirección al dashboard
  await expect(page).toHaveURL('/dashboard');
  await expect(page.locator('h1')).toContainText('Dashboard');
});
```

## Organización de Tests

### Nombres de Archivos

- Archivos de test: `*.spec.ts` (ej: `customer.service.spec.ts`)
- Tests E2E: `*.e2e.spec.ts` (ej: `login.e2e.spec.ts`)

### Estructura de Tests

```
src/
├── app/
│   ├── components/
│   │   └── customer-list/
│   │       ├── customer-list.component.ts
│   │       └── customer-list.component.spec.ts
│   └── services/
│       ├── customer.service.ts
│       └── customer.service.spec.ts
└── e2e/
    └── login.e2e.spec.ts
```

## Consecuencias

### Positivas

- **Calidad de Código:** Los tests detectan bugs temprano y previenen regresiones
- **Confianza:** Refactorización segura con cobertura de tests
- **Documentación:** Los tests sirven como documentación ejecutable
- **Mantenibilidad:** Los tests ayudan a entender el comportamiento del código
- **Colaboración del Equipo:** Patrones de tests consistentes en todo el equipo
- **Integración CI/CD:** Testing automatizado en el pipeline de despliegue

### Negativas

- **Inversión de Tiempo:** Escribir tests requiere tiempo adicional de desarrollo
- **Mantenimiento:** Los tests necesitan actualizarse cuando el código cambia
- **Curva de Aprendizaje:** Los miembros del equipo necesitan aprender patrones de testing
- **Fragilidad de Tests:** Los tests pueden romperse debido a cambios de implementación

### Mitigación

- Comenzar con rutas críticas y áreas de alto riesgo
- Usar generadores de tests y plantillas
- Revisiones regulares de tests para asegurar calidad
- Equilibrar entre tests unitarios e integración
- Enfocarse en testear comportamiento, no implementación

## Ejecutar Tests

```bash
# Ejecutar todos los tests unitarios
npm run test

# Ejecutar tests en modo watch
npm run test -- --watch

# Ejecutar tests con cobertura
npm run test -- --coverage

# Ejecutar tests E2E
npm run test:e2e
```
