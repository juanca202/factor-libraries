# ADR-011: Estrategia de Documentación

**📖 [English](../../en/adr/ADR-011.md) | [Español](./ADR-011.md)**

**Estado:** Aceptado  
**Fecha:** 2025  
**Decisores:** Equipo de Arquitectura

## Contexto

La documentación clara y completa del código es esencial para:
- Incorporar nuevos desarrolladores
- Mantener el código a lo largo del tiempo
- Entender lógica compleja y reglas de negocio
- Generar documentación de API automáticamente
- Asegurar consistencia en todo el código base

Sin una estrategia clara de documentación:
- La documentación puede ser inconsistente o faltante
- Los comentarios pueden estar en diferentes idiomas, causando confusión
- La documentación de API requiere mantenimiento manual
- Las revisiones de código se vuelven más difíciles sin estándares claros de documentación
- Los nuevos miembros del equipo tienen dificultades para entender el código base

Necesitamos un enfoque estandarizado que:
- Asegure que toda la documentación esté en inglés para consistencia
- Genere automáticamente documentación de API a partir de comentarios en el código
- Use un formato estándar (JSDoc/TSDoc) que las herramientas puedan analizar
- Se integre perfectamente en el flujo de trabajo de desarrollo
- Proporcione pautas claras sobre cuándo y cómo documentar el código

## Decisión

Implementaremos una **estrategia integral de documentación** con:

1. **Documentación solo en inglés:** Todos los comentarios de código, anotaciones JSDoc/TSDoc y documentación inline deben estar escritos en inglés
2. **Compodoc:** Usar Compodoc para generar automáticamente documentación de API a partir de comentarios JSDoc/TSDoc
3. **Formato JSDoc/TSDoc:** Todas las APIs públicas, clases, métodos y lógica compleja deben estar documentadas usando anotaciones JSDoc o TSDoc
4. **Estándares de documentación:** Pautas claras sobre qué documentar y cómo

## Implementación

### 1. Requisito de Idioma

**Toda la documentación debe estar en inglés**, incluyendo:
- Comentarios de código
- Anotaciones JSDoc/TSDoc
- Documentación inline
- Descripciones de funciones y métodos
- Descripciones de parámetros
- Descripciones de valores de retorno
- Código de ejemplo en la documentación

**Razonamiento:**
- Asegura consistencia en todo el código base
- Hace el código base accesible para desarrolladores internacionales
- Se alinea con las mejores prácticas de la industria
- Simplifica herramientas y automatización

### 2. Configuración de Compodoc

Compodoc está configurado para generar automáticamente documentación de API a partir de comentarios JSDoc/TSDoc.

**Configuración:** `compodoc.json` o scripts en `package.json`

**Uso:**
```bash
npm run compodoc        # Generar documentación
npm run compodoc:serve  # Generar y servir documentación localmente
```

**Salida:** La documentación se genera en el directorio `documentation/`

### 3. Formato JSDoc/TSDoc

Usar **TSDoc** (específico de TypeScript) para archivos TypeScript y **JSDoc** para archivos JavaScript.

#### Formato TSDoc (Recomendado para TypeScript)

TSDoc es el formato recomendado para proyectos TypeScript ya que proporciona mejor conocimiento de tipos e integración con herramientas TypeScript.

**Ejemplo Básico:**
```typescript
/**
 * Calculates the total price including tax.
 *
 * @param basePrice - The base price before tax
 * @param taxRate - The tax rate as a decimal (e.g., 0.10 for 10%)
 * @returns The total price including tax
 *
 * @example
 * ```typescript
 * const total = calculateTotalPrice(100, 0.10);
 * console.log(total); // 110
 * ```
 */
export function calculateTotalPrice(basePrice: number, taxRate: number): number {
  return basePrice * (1 + taxRate);
}
```

### 4. Estándares de Documentación

#### Qué Documentar

**Debe Documentarse:**
- Todas las APIs públicas (clases, funciones, interfaces, tipos exportados)
- Algoritmos complejos o lógica de negocio
- Comportamiento de código no obvio
- Inputs y outputs públicos de componentes
- Métodos de servicios y sus parámetros
- Validadores y pipes personalizados
- Funciones de utilidad con lógica no trivial

**Debería Documentarse:**
- Métodos privados con lógica compleja
- Soluciones temporales o workarounds
- Consideraciones de rendimiento
- Limitaciones conocidas o casos extremos

**Opcional:**
- Getters/setters simples
- Código autoexplicativo
- Hooks de ciclo de vida estándar de Angular (a menos que tengan comportamiento personalizado)

#### Etiquetas de Documentación

**Etiquetas Comunes TSDoc/JSDoc:**

- `@param` - Descripción de parámetro
- `@returns` o `@return` - Descripción del valor de retorno
- `@throws` o `@throws` - Excepciones que pueden lanzarse
- `@example` - Ejemplos de uso
- `@remarks` - Notas adicionales
- `@deprecated` - Marcar APIs deprecadas
- `@since` - Versión en la que se introdujo la API
- `@see` - Referencia a documentación relacionada
- `@internal` - Marcar APIs internas (no incluidas en docs públicas)

**Ejemplo con Múltiples Etiquetas:**
```typescript
/**
 * Validates an email address format.
 *
 * @param email - The email address to validate
 * @returns True if the email format is valid, false otherwise
 * @throws {TypeError} If email is not a string
 *
 * @example
 * ```typescript
 * isValidEmail('user@example.com'); // true
 * isValidEmail('invalid-email');     // false
 * ```
 *
 * @remarks
 * This validator checks format only, not domain existence.
 * For production use, consider additional validation.
 *
 * @since 1.0.0
 */
export function isValidEmail(email: string): boolean {
  if (typeof email !== 'string') {
    throw new TypeError('Email must be a string');
  }
  const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  return emailRegex.test(email);
}
```

### 5. Integración con Compodoc

Compodoc escanea automáticamente el código base en busca de comentarios JSDoc/TSDoc y genera:
- Documentación de API
- Documentación de componentes
- Documentación de servicios
- Documentación de módulos
- Documentación de rutas

**Ejemplo de Configuración:**
```json
{
  "name": "angular-base-project",
  "compodoc": {
    "output": "documentation",
    "theme": "gitbook",
    "includes": "src/**/*.ts",
    "includesName": "API Documentation"
  }
}
```

**La Documentación Generada Incluye:**
- Documentación de clases e interfaces
- Descripciones de métodos y propiedades
- Información de tipos de parámetros y retorno
- Ejemplos de uso
- Gráficos de dependencias de módulos
- Visualización del árbol de componentes

## Consecuencias

### Positivas

- **Consistencia:** Toda la documentación en inglés asegura comprensión uniforme
- **Automatización:** Compodoc genera automáticamente documentación de API
- **Mantenibilidad:** El formato estándar facilita la actualización de documentación
- **Experiencia del Desarrollador:** La documentación clara ayuda en la incorporación y desarrollo
- **Seguridad de Tipos:** TSDoc proporciona mejor integración con TypeScript
- **Soporte de Herramientas:** JSDoc/TSDoc es ampliamente soportado por IDEs y herramientas
- **Profesionalismo:** Un código base bien documentado refleja estándares profesionales

### Negativas

- **Costo Inicial:** Requiere tiempo para documentar código existente
- **Mantenimiento:** La documentación debe mantenerse sincronizada con cambios en el código
- **Curva de Aprendizaje:** Los miembros del equipo necesitan aprender convenciones JSDoc/TSDoc
- **Requisito de Inglés:** Puede ser desafiante para hablantes no nativos de inglés

### Mitigación

- **Adopción Gradual:** Documentar código nuevo primero, actualizar código existente gradualmente
- **Revisiones de Código:** Incluir revisión de documentación en el proceso de pull request
- **Plantillas:** Proporcionar plantillas JSDoc/TSDoc para patrones comunes
- **Capacitación:** Ofrecer capacitación en mejores prácticas de documentación
- **Automatización:** Usar reglas de linting para hacer cumplir documentación en APIs públicas
- **Ejemplos:** Mantener una biblioteca de ejemplos de código bien documentados

## Mejores Prácticas

1. **Escribir documentación mientras se codifica:** No dejarlo para después
2. **Mantenerla concisa:** Enfocarse en qué, por qué y cómo, no en detalles obvios
3. **Usar ejemplos:** Los ejemplos de código aclaran el uso mejor que las descripciones
4. **Actualizar con el código:** Al cambiar código, actualizar documentación también
5. **Revisar documentación:** Incluir documentación en revisiones de código
6. **Usar nombres significativos:** Los buenos nombres reducen la necesidad de documentación extensa
7. **Documentar el "por qué":** Explicar el razonamiento, no solo lo que hace el código
