# ADR-004: Reglas de Desarrollo Asistido por IA

**📖 [English](../../en/adr/ADR-004.md) | [Español](./ADR-004.md)**

**Estado:** Aceptado  
**Fecha:** 2025  
**Decisores:** Equipo de Arquitectura

## Contexto

A medida que las herramientas de desarrollo asistido por IA se vuelven más prevalentes, existe el riesgo de que:
- El código generado puede no seguir las convenciones del proyecto
- Las decisiones arquitectónicas pueden ser pasadas por alto o violadas
- La calidad y consistencia del código pueden degradarse
- Diferentes desarrolladores usando herramientas de IA pueden producir código inconsistente
- Las mejores prácticas y patrones pueden no aplicarse consistentemente

Sin reglas y documentación claras, las herramientas de IA no pueden asistir efectivamente a los desarrolladores mientras mantienen la integridad arquitectónica del proyecto y los estándares de calidad del código.

## Decisión

Estableceremos reglas de desarrollo exhaustivas documentadas en `docs/` que pueden ser cargadas automáticamente por herramientas de desarrollo asistido por IA (por ejemplo, a través de archivos de configuración como `.cursor/rules.json`) para asegurar:

1. Generación de código consistente alineada con la arquitectura del proyecto
2. Aplicación automática de decisiones arquitectónicas
3. Adherencia a la Guía de Estilo de Angular y convenciones del proyecto
4. Calidad y mantenibilidad del código generado por IA

## Implementación

### Estructura de Documentación de Reglas

Todas las decisiones arquitectónicas y reglas de estilo están documentadas en `docs/` como archivos Markdown. Las herramientas de desarrollo asistido por IA pueden cargar automáticamente estos archivos a través de archivos de configuración (por ejemplo, `.cursor/rules.json` para Cursor):

```json
[
  {
    "name": "Base architecture docs",
    "description": "Load all architecture documentation for better suggestions.",
    "priority": "high",
    "match": ["**/*"],
    "paths": ["docs/**/*.md"]
  }
]
```

La estructura de documentación en `docs/` incluye:

```
docs/
├── README.md                # Arquitectura general y diagramas C4
├── css-styling.md                 # Guías de CSS y Tailwind
└── adr/                          # Registros de Decisiones Arquitectónicas
    ├── ADR-001.md
    ├── ADR-002.md
    ├── ADR-003.md
    ├── ADR-004.md
    └── ...
```

**Nota:** Hay una distinción entre:
- **Documentación de arquitectura** (`docs/*.md`): Fuente de verdad para reglas arquitectónicas, puede ser cargada por herramientas de desarrollo asistido por IA vía archivos de configuración
- **Reglas específicas del IDE** (por ejemplo, `.cursor/rules/*.mdc` para Cursor): Configuración adicional específica del IDE y reglas siempre aplicadas

- Priorizar utilidades de Tailwind CSS para layout, espaciado y tipografía

## Cómo se Aplican las Reglas

### Aplicación Automática vía Archivos de Configuración

Las herramientas de desarrollo asistido por IA pueden cargar automáticamente toda la documentación de `docs/` a través de archivos de configuración. Por ejemplo, Cursor usa `.cursor/rules.json`:

```json
{
  "name": "Base architecture docs",
  "description": "Load all architecture documentation for better suggestions.",
  "priority": "high",
  "match": ["**/*"],
  "paths": ["docs/**/*.md"]
}
```

Esta configuración asegura que:

1. **Conciencia de Contexto:** La herramienta de IA lee todos los archivos Markdown de `docs/` automáticamente
2. **Generación de Código:** El código generado sigue patrones y convenciones documentados en la documentación de arquitectura
3. **Sugerencias:** Las sugerencias de IA se alinean con decisiones arquitectónicas documentadas en `docs/`
4. **Refactorización:** Las sugerencias de refactorización mantienen la estructura del proyecto como se define en la documentación
5. **Prioridad:** Alta prioridad asegura que estas reglas siempre se consideren durante la generación de código

### Ejemplo: Generación de Código

**Solicitud del Desarrollador:**
> "Create a new sales list component"

**Respuesta de IA (siguiendo reglas):**
```typescript
// ✅ El código generado sigue todas las reglas
@Component({
  selector: 'ft-sales-list',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  imports: [CommonModule, SaleItemComponent],
  template: `
    <div class="flex flex-col gap-4 p-6">
      <h2 class="text-2xl font-bold">Sales</h2>
      @for (sale of sales(); track sale.id) {
        <app-sale-item [sale]="sale" />
      }
    </div>
  `
})
export class SalesListComponent {
  private salesService = inject(SalesService);
  
  sales = this.salesService.sales;
}
```

**Reglas Aplicadas:**
- ✅ Componente standalone
- ✅ Detección de cambios OnPush
- ✅ `inject()` en lugar de constructor
- ✅ Signals para estado
- ✅ Flujo de control nativo (`@for`)
- ✅ Utilidades de Tailwind para estilos
- ✅ Prefijo de selector apropiado (`ft-`)

## Beneficios

### 1. Consistencia

- **Estilo de Código Uniforme:** Todo el código generado por IA sigue los mismos patrones
- **Estructura Predecible:** Los desarrolladores saben qué esperar de las sugerencias de IA
- **Tiempo de Revisión Reducido:** Menos tiempo gastado en problemas de estilo y convención

### 2. Calidad

- **Mejores Prácticas:** La IA aplica automáticamente las mejores prácticas de Angular y del proyecto
- **Cumplimiento Arquitectónico:** El código generado respeta las decisiones arquitectónicas
- **Prevención de Errores:** Los errores comunes se evitan mediante la aplicación de reglas

### 3. Onboarding

- **Aprendizaje Más Rápido:** Los nuevos desarrolladores ven patrones consistentes inmediatamente
- **Auto-documentación:** Las reglas sirven como documentación viva
- **Preguntas Reducidas:** Las guías claras reducen la necesidad de aclaración

### 4. Mantenimiento

- **Documentación Viva:** Las reglas son ejecutables y siempre actualizadas
- **Actualizaciones Fáciles:** Los cambios arquitectónicos se reflejan en las reglas inmediatamente
- **Control de Versiones:** Las reglas están versionadas con el codebase

## Mantenimiento de Reglas

### Agregar Nuevas Reglas

1. **Identificar Necesidad:** Reconocer un patrón o convención que debe estandarizarse
2. **Documentar Regla:** Agregar regla al archivo Markdown apropiado en `docs/` (ej: `docs/css-styling.md`) o crear un nuevo ADR en `docs/adr/` si representa una decisión arquitectónica significativa
3. **Proporcionar Ejemplos:** Incluir ejemplos correctos e incorrectos en la documentación
4. **Actualizar ADRs:** Si la regla representa una decisión arquitectónica significativa, crear o actualizar un ADR en `docs/adr/`
5. **Verificar Carga:** Asegurar que la nueva documentación se carga automáticamente verificando que el archivo de configuración del IDE (por ejemplo, `.cursor/rules.json` para Cursor) incluye `docs/**/*.md`
6. **Revisión del Equipo:** Compartir nuevas reglas con el equipo

### Actualizar Reglas Existentes

1. **Identificar Cambio:** Reconocer cuando una regla necesita modificación
2. **Actualizar Documentación:** Modificar la regla en el archivo apropiado en `docs/`
3. **Actualizar Ejemplos:** Asegurar que los ejemplos reflejen la nueva regla
4. **Actualizar ADRs:** Si el cambio representa una decisión significativa, actualizar el ADR relevante
5. **Comunicar:** Notificar al equipo de cambios en las reglas
6. **Refactorizar Código:** Actualizar código existente para coincidir con las nuevas reglas si es necesario

**Nota:** Dado que `rules.json` usa el patrón `docs/**/*.md`, cualquier archivo Markdown nuevo o actualizado en `docs/` se incluye automáticamente sin modificar `rules.json`.

## Estándar de Documentación de Decisiones Arquitectónicas

Todas las decisiones arquitectónicas significativas siguen el formato de **Registro de Decisión Arquitectónica (ADR)**, que es el estándar para documentar decisiones arquitectónicas en este proyecto.

### Formato ADR

Cada ADR sigue una estructura consistente:

- **Título:** Título claro y descriptivo (ej: `ADR-XXX: Nombre de la Decisión`)
- **Estado:** Estado actual (Aceptado, Propuesto, Deprecado, etc.)
- **Fecha:** Cuándo se tomó la decisión
- **Decisores:** Quién tomó la decisión
- **Contexto:** La situación y problema que llevó a esta decisión
- **Decisión:** La elección arquitectónica que se tomó
- **Implementación:** Guías de implementación detalladas y ejemplos
- **Consecuencias:** Impactos positivos y negativos de la decisión
- **Referencias:** Enlaces a documentación relacionada y recursos

### Ubicación de los ADRs

Todos los ADRs están ubicados en `docs/en/adr/` (inglés) y `docs/es/adr/` (español), siguiendo la convención de nombres `ADR-XXX.md` donde XXX es un número secuencial.

### Puntos Clave

- **Formato Estándar:** Todas las decisiones arquitectónicas usan el formato ADR para consistencia
- **Control de Versiones:** Los ADRs están versionados con el código base
- **Bilingüe:** Los ADRs se mantienen tanto en inglés como en español
- **Contexto de IA:** Los ADRs sirven tanto como documentación legible por humanos como contexto para herramientas de desarrollo asistido por IA
- **Documentación Viva:** Los ADRs se actualizan a medida que las decisiones evolucionan

Para más información sobre ADRs, ver [Registros de Decisiones Arquitectónicas (ADRs)](./README.md).

## Consecuencias

### Positivas

- **Consistencia:** Todo el código generado por IA sigue los estándares del proyecto
- **Calidad:** Aplicación automática de mejores prácticas
- **Velocidad:** Desarrollo más rápido con asistencia de IA
- **Onboarding:** Los nuevos desarrolladores aprenden patrones a través de sugerencias de IA
- **Documentación:** Las reglas sirven como documentación ejecutable
- **Escalabilidad:** Fácil agregar nuevas reglas a medida que el proyecto evoluciona

### Negativas

- **Configuración Inicial:** Requiere tiempo para documentar todas las reglas exhaustivamente
- **Mantenimiento:** Las reglas necesitan mantenerse actualizadas con la evolución de Angular y del proyecto
- **Rigidez:** Puede limitar la flexibilidad en casos extremos (aunque generalmente es positivo)
- **Dependencia de Herramientas:** Depende de la capacidad de la herramienta de IA para leer y aplicar reglas

### Mitigación

- Comenzar con reglas esenciales y expandir gradualmente
- Revisión y actualización regular de reglas
- Equilibrio entre estrictez y flexibilidad
- Documentar excepciones y casos extremos
- Retroalimentación regular del equipo sobre la efectividad de las reglas
