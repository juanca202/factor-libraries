# ADR-009: Calidad de Código y Herramientas

**📖 [English](../../en/adr/ADR-009.md) | [Español](./ADR-009.md)**

**Estado:** Aceptado  
**Fecha:** 2025  
**Decisores:** Equipo de Arquitectura

## Contexto

Mantener la calidad del código y la consistencia en un equipo requiere herramientas automatizadas y estándares claros. Sin herramientas adecuadas:

- Inconsistencias de estilo de código entre desarrolladores
- Bugs y problemas descubiertos tarde en el desarrollo
- Revisiones de código difíciles enfocadas en estilo en lugar de lógica
- Mensajes de commit inconsistentes haciendo difícil navegar el historial
- Formateo y linting manual consume tiempo y es propenso a errores
- No hay aplicación automatizada de estándares de codificación

Necesitamos un conjunto exhaustivo de herramientas que:
- Aplique estilo de código consistente automáticamente
- Detecte errores y bugs potenciales temprano
- Valide mensajes de commit para mejor historial del proyecto
- Se integre perfectamente en el flujo de trabajo de desarrollo
- Requiera intervención manual mínima

## Decisión

Usaremos una **cadena de herramientas de calidad de código exhaustiva** con:

1. **ESLint:** Linting de código y detección de errores
2. **Prettier:** Formateo automático de código
3. **Husky:** Git hooks para validación pre-commit y commit-msg
4. **lint-staged:** Ejecutar linters solo en archivos staged
5. **Commitlint:** Hacer cumplir el formato Conventional Commits
6. **TypeScript:** Verificación estricta de tipos
7. **Compodoc:** Generación automática de documentación API

Todas las herramientas están configuradas para trabajar juntas y hacer cumplir estándares automáticamente.

## Implementación

### Stack de Herramientas

#### 1. ESLint

**Propósito:** Análisis estático de código y linting

**Configuración:** `eslint.config.js`

**Características Clave:**
- Reglas específicas de TypeScript
- Reglas específicas de Angular vía `@angular-eslint`
- Linting de templates para templates de Angular
- Integración con Prettier

**Uso:**
```bash
# Verificar errores de linting
npm run lint

# Auto-corregir errores de linting
npm run lint:fix
```

**Reglas Clave:**
- Aplicación de modo estricto de TypeScript
- Mejores prácticas de Angular
- Detección de variables no usadas
- Ordenamiento de imports
- Límites de complejidad de código

#### 2. Prettier

**Propósito:** Formateo automático de código

**Configuración:** `.prettierrc`

**Configuración:**
```json
{
  "printWidth": 100,
  "singleQuote": true,
  "trailingComma": "none",
  "overrides": [
    {
      "files": "*.html",
      "options": {
        "parser": "angular"
      }
    }
  ]
}
```

**Uso:**
```bash
# Formatear todos los archivos
npm run prettier
```

**Formatea:**
- Archivos TypeScript/JavaScript
- Templates HTML
- Archivos SCSS/CSS
- Archivos JSON
- Archivos Markdown

#### 3. Husky

**Propósito:** Git hooks para verificaciones automatizadas

**Hooks Configurados:**

**pre-commit** (`.husky/pre-commit`):
```bash
npx lint-staged --no-stash
```
- Ejecuta linting y formateo en archivos staged
- Previene commits con errores de linting
- Auto-corrige problemas cuando es posible

**commit-msg** (`.husky/commit-msg`):
```bash
npx --no -- commitlint --edit "$1"
```
- Valida formato de mensaje de commit
- Hace cumplir el estándar Conventional Commits

#### 4. lint-staged

**Propósito:** Ejecutar linters solo en archivos staged

**Configuración:** `package.json`
```json
{
  "lint-staged": {
    "src/**/*.{ts,js,html,scss,json,md}": [
      "prettier --write",
      "eslint --fix"
    ]
  }
}
```

**Beneficios:**
- Verificaciones pre-commit más rápidas
- Solo procesa archivos cambiados
- Auto-corrige problemas antes del commit

#### 5. Commitlint

**Propósito:** Hacer cumplir convenciones de mensajes de commit

**Configuración:** `commitlint.config.js`
```javascript
module.exports = {
  extends: ['@commitlint/config-conventional']
};
```

**Formato Conventional Commits:**
```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

**Tipos Permitidos:**
- `feat`: Nueva funcionalidad
- `fix`: Corrección de bug
- `docs`: Cambios en documentación
- `style`: Cambios de estilo de código (formateo, etc.)
- `refactor`: Refactorización de código
- `test`: Agregar o actualizar tests
- `chore`: Tareas de mantenimiento

**Ejemplos:**
```bash
# ✅ Commits válidos
feat(auth): add Google login
fix(api): correct error handling in user endpoint
docs(readme): update installation instructions
style(components): apply prettier formatting
refactor(core): optimize notification service
test(auth): add login flow tests
chore(deps): update Angular to v20

# ❌ Commits inválidos
Added new feature
fix bug
update code
```

#### 6. TypeScript

**Propósito:** Seguridad de tipos y detección de errores en tiempo de compilación

**Configuración:** `tsconfig.json` con modo estricto

**Configuraciones Clave:**
- `strict: true` - Habilita todas las opciones de verificación estricta de tipos
- `noImplicitAny: true` - Previene tipos any implícitos
- `strictNullChecks: true` - Verificación de null y undefined
- `noUnusedLocals: true` - Error en variables locales no usadas
- `noUnusedParameters: true` - Error en parámetros no usados

#### 7. Compodoc

**Propósito:** Generación automática de documentación API

**Uso:**
```bash
# Generar documentación
npm run compodoc:build

# Generar y servir documentación
npm run compodoc:build-and-serve

# Servir documentación existente
npm run compodoc:serve
```

**Características:**
- Extrae comentarios JSDoc/TSDoc
- Genera documentación interactiva
- Incluye documentación de componentes, servicios y módulos

## Integración de Flujo de Trabajo

### Flujo de Trabajo Pre-Commit

1. Desarrollador hace stage de archivos: `git add .`
2. Desarrollador hace commit: `git commit -m "feat: add new feature"`
3. **Hook pre-commit de Husky se activa:**
   - `lint-staged` se ejecuta en archivos staged
   - Prettier formatea archivos
   - ESLint verifica y auto-corrige problemas
   - Si quedan errores, el commit se bloquea
4. **Hook commit-msg de Husky se activa:**
   - Commitlint valida formato de mensaje
   - Si es inválido, el commit se rechaza con mensaje de error
5. Si todas las verificaciones pasan, el commit tiene éxito

### Flujo de Trabajo de Desarrollo

```bash
# 1. Hacer cambios en el código
# Editar archivos...

# 2. Hacer stage de cambios
git add .

# 3. Commit (los hooks se ejecutan automáticamente)
git commit -m "feat(component): add new feature"

# Si existen errores de linting, se auto-corrigen
# Si el mensaje de commit es inválido, verás un error
```

### Verificaciones de Calidad Manuales

```bash
# Ejecutar todas las verificaciones de linting
npm run lint

# Auto-corregir problemas de linting
npm run lint:fix

# Formatear todo el código
npm run prettier

# Ejecutar tests
npm run test
```

Ubicados en `.husky/`:
- `pre-commit`: Ejecuta lint-staged
- `commit-msg`: Valida mensajes de commit

## Mejores Prácticas

### 1. Dejar que las Herramientas Hagan el Trabajo

```bash
# ✅ Correcto - dejar que Prettier formatee
npm run prettier

# ❌ Incorrecto - formateo manual
# Ajustar manualmente espaciado, comillas, etc.
```

### 2. Corregir Errores de Linting Antes de Hacer Commit

```bash
# ✅ Correcto - corregir problemas primero
npm run lint:fix
git add .
git commit -m "fix: resolve linting errors"

# ❌ Incorrecto - commit con errores
git commit -m "feat: new feature"  # Será bloqueado por hooks
```

### 3. Usar Conventional Commits

```bash
# ✅ Correcto - sigue la convención
git commit -m "feat(auth): add password reset"
git commit -m "fix(api): handle null response"

# ❌ Incorrecto - no sigue la convención
git commit -m "added password reset"
git commit -m "fixed bug"
```

### 4. Ejecutar Verificaciones de Calidad Localmente

```bash
# ✅ Correcto - verificar antes de hacer push
npm run lint
npm run test
git push

# ❌ Incorrecto - depender solo de CI
git push  # Puede fallar en CI
```

### 5. Mantener Configuración Actualizada

- Actualizar reglas de ESLint a medida que Angular evoluciona
- Mantener configuración de Prettier consistente
- Revisar y actualizar reglas de commitlint según sea necesario

## Consecuencias

### Positivas

- **Consistencia:** Todo el código sigue el mismo estilo automáticamente
- **Calidad:** Errores detectados temprano en el desarrollo
- **Productividad:** Menos tiempo gastado en revisiones de código por problemas de estilo
- **Historial:** Historial de commits limpio y buscable
- **Onboarding:** Los nuevos desarrolladores siguen estándares automáticamente
- **Mantenibilidad:** El código consistente es más fácil de mantener
- **Automatización:** Intervención manual mínima requerida

### Negativas

- **Configuración Inicial:** Requiere tiempo de configuración y setup
- **Curva de Aprendizaje:** El equipo necesita aprender configuraciones de herramientas
- **Rechazo de Commits:** Los commits inválidos se rechazan (aunque esto es positivo)
- **Dependencia CI/CD:** Depende de que las herramientas funcionen correctamente

### Mitigación

- Documentación clara de todas las herramientas y configuraciones
- Entrenamiento del equipo en uso de herramientas
- Actualizaciones regulares de configuraciones de herramientas
- Pipeline CI/CD como validación de respaldo

## Integración CI/CD

Estas herramientas también deben integrarse en el pipeline CI/CD:

```yaml
# Ejemplo de pasos CI/CD
- name: Lint code
  run: npm run lint

- name: Format check
  run: npm run prettier -- --check

- name: Run tests
  run: npm run test

- name: Build
  run: npm run build
```

## Actualizar Herramientas

Al actualizar herramientas:

1. **Actualizar dependencias:** `npm update`
2. **Probar localmente:** Ejecutar todas las verificaciones de calidad
3. **Actualizar configuraciones:** Ajustar reglas según sea necesario
4. **Documentar cambios:** Actualizar este ADR si hay cambios significativos
5. **Comunicación del equipo:** Notificar al equipo de cambios que rompen compatibilidad
