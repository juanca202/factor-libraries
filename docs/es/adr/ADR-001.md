# ADR-001: SeparaciГіn de Responsabilidades - Core, Shared y Features

**рҹ“– [English](../../en/adr/ADR-001.md) | [EspaГұol](./ADR-001.md)**

**Estado:** Aceptado  
**Fecha:** 2025
**Decisores:** Equipo de Arquitectura

## Contexto

A medida que la aplicaciГіn crece, se vuelve crucial mantener una separaciГіn clara de concerns para asegurar:
- ReutilizaciГіn de cГіdigo entre diferentes features
- Desarrollo y testing independiente de features
- FГЎcil mantenimiento y refactorizaciГіn
- PrevenciГіn de dependencias circulares
- ComprensiГіn clara de dГіnde pertenece el cГіdigo

Sin una estructura bien definida, los desarrolladores podrГӯan colocar cГіdigo en ubicaciones inapropiadas, llevando a acoplamiento estrecho, dificultad en testing y desafГӯos al escalar la aplicaciГіn.

## DecisiГіn

Organizaremos la aplicaciГіn en cuatro capas distintas con responsabilidades claras:

### Capa Core (`src/app/core/`)

Contiene infraestructura y lГіgica global que se usa en toda la aplicaciГіn:

- **Services:** Servicios singleton globales (ej: `HttpClient`, `Logger`, `NotificationService`)
- **Guards:** Guards de ruta para autenticaciГіn y autorizaciГіn
- **Interceptors:** Interceptores HTTP para manejo de request/response
- **Models:** Modelos de datos globales e interfaces usadas entre features
- **Enums:** Enumeraciones compartidas (ej: `UserRole`, `HttpStatus`)
- **Constants:** ConfiguraciГіn global y constantes
- **Utils:** Funciones helper para operaciones comunes
- **Pipes:** Pipes de transformaciГіn comunes
- **Directives:** Directivas globales
- **Components:** Componentes UI globales (ej: layout principal, pГЎginas de error)

**Principio clave:** Core no debe depender de Shared, Cross ni Features.

### Capa Shared (`src/app/shared/`)

Contiene componentes UI reutilizables y utilidades que pueden ser usadas por mГәltiples features:

- **Components:** Componentes de presentaciГіn reutilizables (botones, modales, tarjetas, campos de formulario)
- **Directives:** Directivas de atributo o estructurales compartidas
- **Pipes:** Pipes de formateo UI (ej: currency, formateo de fechas)
- **Validators:** Validadores de formularios comunes
- **Types/Interfaces:** Definiciones de tipos compartidas para componentes UI

**Principio clave:** Shared no debe depender de Features, pero puede depender de Core y Cross.

### Capa Cross (`src/app/cross/{dominio}/`)

Contiene capacidades de dominio transversales que aplican a mГәltiples features y encapsulan reglas de negocio, orquestaciones o integraciones que no pertenecen a Shared porque no son utilidades de UI:

- **Services:** Servicios de negocio transversales (ej: `AuthService`, `SessionService`)
- **Repositories:** Acceso a API o almacenamiento reutilizado por diferentes features
- **Managers:** Coordinadores de flujos compartidos (inicio de sesiГіn, inicializaciГіn de analytics)
- **Models:** Modelos de dominio reutilizados mГЎs allГЎ de la infraestructura bГЎsica
- **Policies/Guards:** Reglas aplicables en mГәltiples contextos (feature toggles, verificaciones de capacidades)
- **Pipes/Validators:** Helpers especГӯficos del dominio que no encajan en Shared

**Principio clave:** Cross solo puede depender de Core.

### Capa Features (`src/app/features/{feature-name}/`)

Contiene funcionalidad especГӯfica del dominio organizada por feature:

- **Components:** Componentes especГӯficos de la feature
- **Services:** Servicios de lГіgica de negocio para la feature
- **Repositories:** Capa de acceso a datos para comunicaciГіn con API
- **Managers:** GestiГіn de estado o orquestaciГіn de lГіgica de negocio
- **Models:** Modelos de datos especГӯficos de la feature
- **Guards:** Guards de ruta especГӯficos de la feature
- **Pipes:** Pipes de transformaciГіn especГӯficos de la feature
- **Directives:** Directivas especГӯficas de la feature
- **Utils:** Funciones helper especГӯficas de la feature
- **Enums:** Enumeraciones especГӯficas de la feature
- **Constants:** Constantes especГӯficas de la feature
- **Routes:** Rutas de la feature definidas en `{feature-name}-routes.ts`

**Principios clave:**
- Cada feature es independiente y autocontenida
- Las features pueden depender de Core, Shared y Cross, pero no de otras Features
- Las features no deben importar directamente de otras features

### Dependencias Permitidas

вң… **Feature вҶ’ Core:** Permitido
```typescript

```

вң… **Feature вҶ’ Shared:** Permitido
```typescript

```

вң… **Feature вҶ’ Cross:** Permitido
```typescript

```

вң… **Shared вҶ’ Core:** Permitido
```typescript

```

вң… **Shared вҶ’ Cross:** Permitido
```typescript

```

вң… **Cross вҶ’ Core:** Permitido

### Dependencias Prohibidas

вқҢ **Core вҶ’ Shared:** No permitido
```typescript
// вқҢ NO HACER: core/services/logger.service.ts

```

вқҢ **Core вҶ’ Feature:** No permitido
```typescript
// вқҢ NO HACER: core/guards/auth.guard.ts

```

вқҢ **Core вҶ’ Cross:** No permitido
```typescript
// вқҢ NO HACER: core/interceptors/http-error.interceptor.ts

```

вқҢ **Shared вҶ’ Feature:** No permitido
```typescript
// вқҢ NO HACER: shared/components/button/button.component.ts

```

вқҢ **Cross вҶ’ Shared o Feature:** No permitido


## DefiniciГіn de Rutas

Cada feature define sus propias rutas en un archivo `{feature-name}-routes.ts` dentro de `src/app/features/{feature-name}/`:

```typescript
// app/features/sales/sales-routes.ts
import { Routes } from '@angular/router';
import { SalesListComponent } from './components/sales-list/sales-list.component';
import { SaleDetailComponent } from './components/sale-detail/sale-detail.component';

export const salesRoutes: Routes = [
  {
    path: '',
    component: SalesListComponent,
  },
  {
    path: ':id',
    component: SaleDetailComponent,
  },
];
```

Luego, estas rutas se importan en las rutas principales de la app:

```typescript
// app.routes.ts
import { salesRoutes } from './features/sales/sales-routes';

export const routes: Routes = [
  {
    path: 'sales',
    loadChildren: () => import('./features/sales/sales-routes').then(m => m.salesRoutes),
  },
  // ... otras rutas
];
```

## Consecuencias

### Positivas

- **SeparaciГіn clara de concerns:** Los desarrolladores saben exactamente dГіnde colocar nuevo cГіdigo
- **ReutilizaciГіn:** Componentes Shared y servicios Core pueden usarse entre features
- **Testabilidad:** Las features pueden probarse independientemente con dependencias Core/Shared mockeadas
- **Escalabilidad:** Nuevas features pueden agregarse sin afectar las existentes
- **Mantenibilidad:** Cambios en una feature no impactan otras
- **Onboarding:** Nuevos desarrolladores pueden entender rГЎpidamente la estructura del cГіdigo

### Negativas

- **Complejidad inicial de setup:** Requiere disciplina para mantener la estructura
- **Posible duplicaciГіn de cГіdigo:** LГіgica similar podrГӯa existir en diferentes features (aceptable si es verdaderamente especГӯfica de la feature)
- **Curva de aprendizaje:** Los nuevos miembros del equipo necesitan entender las reglas de dependencias

### MitigaciГіn

- Documentar la estructura claramente (este ADR)
- Usar aliases de ruta (`@/core`, `@/shared`) para hacer los imports claros
- Hacer cumplir las reglas mediante linting (las reglas de ESLint pueden prevenir imports prohibidos)
- Revisiones de cГіdigo regulares para asegurar cumplimiento
