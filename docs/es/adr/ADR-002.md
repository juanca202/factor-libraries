# ADR-002: Adopción de la Guía de Estilo Oficial de Angular

**📖 [English](../../en/adr/ADR-002.md) | [Español](./ADR-002.md)**

**Estado:** Aceptado  
**Fecha:** 2025  
**Decisores:** Equipo de Arquitectura

## Contexto

A medida que Angular evoluciona y el equipo crece, mantener la consistencia del código se vuelve desafiante. Diferentes desarrolladores pueden tener diferentes estilos de codificación, llevando a:
- Patrones de código inconsistentes en todo el codebase
- Dificultad en las revisiones de código
- Onboarding más lento para nuevos desarrolladores
- Desafíos de mantenimiento
- Posibles problemas de rendimiento por patrones no óptimos

El equipo de Angular proporciona una [Guía de Estilo Oficial de Angular](https://angular.dev/style-guide) que representa las mejores prácticas y patrones recomendados para aplicaciones Angular.

## Decisión

Adoptaremos y seguiremos estrictamente la Guía de Estilo Oficial de Angular para asegurar consistencia, mantenibilidad y alineación con las mejores prácticas de Angular.

Todo el código en este proyecto debe conformarse a la Guía de Estilo de Angular, y estas reglas están documentadas en `.cursor/rules/` para ser aplicadas automáticamente por herramientas de desarrollo asistido por IA.

## Aspectos Clave Aplicados

### 1. Convenciones de Nombres de Archivos

**Regla:** Usar kebab-case (guiones) para separar palabras en nombres de archivos. Los sufijos obligatorios también deben escribirse con guiones (por ejemplo, `-guard`, `-validator`).

**Sufijos obligatorios:** Solo los Guards, Pipes, Resolvers, Interceptors y Validators deben incluir su sufijo tanto en la clase como en el nombre del archivo (por ejemplo, `AuthGuard` reside en `auth-guard.ts`). El resto de artefactos (componentes, servicios, modelos, utilidades, etc.) debe evitar sufijos artificiales como `-component` o `-model`.

**Ejemplos:**
```typescript
// ✅ Correcto
user-profile.ts
sales-list.ts
http-error-interceptor.ts
email-validator.ts

// ❌ Incorrecto
userProfile.ts
salesList.ts
httpError-interceptor.ts
emailValidator.ts
```

### 2. Convenciones de Nombres de Clases

**Regla:** Usar PascalCase para clases, interfaces y tipos.

**Estándar de sufijos:** Las clases Guard, Pipe, Resolver, Interceptor y Validator deben terminar con su sufijo correspondiente y el nombre del archivo debe reflejarlo (por ejemplo, `auth-guard.ts`, `format-currency-pipe.ts`).

**Ejemplos:**
```typescript
// ✅ Correcto
export class UserProfile {}
export interface UserModel {}
export type UserRole = 'admin' | 'user';
export class AuthGuard {}
export class FormatCurrencyPipe {}
export class LoadCustomerResolver {}
export class HttpErrorInterceptor {}

// ❌ Incorrecto
export class userProfile {}
export interface userModel {}
export class Auth {}
export class FormatCurrency {}
export class LoadCustomer {}
export class HttpError {}
```

### 3. Estructura del Proyecto

**Regla:** Organizar código por feature, no por tipo de archivo.

**Ejemplo de Estructura:**
```
// ✅ Correcto - organizado por feature
sales/
  ├── components/
  ├── services/
  ├── models/
  └── sales-routes.ts

// ❌ Incorrecto - organizado por tipo
components/
  ├── sales-list.ts
  ├── sales-detail.ts
services/
  ├── sales-service.ts
models/
  └── sale.ts
```

### 4. Inyección de Dependencias

**Regla:** Preferir la función `inject()` sobre la inyección por constructor.

**Ejemplos:**
```typescript
// ✅ Correcto - usando inject()
import { inject } from '@angular/core';
import { HttpClient } from '@angular/common/http';

export class SalesService {
  private http = inject(HttpClient);
  private logger = inject(Logger);
}

// ❌ Incorrecto - inyección por constructor
export class SalesService {
  constructor(
    private http: HttpClient,
    private logger: Logger
  ) {}
}
```

### 5. Inputs y Outputs de Componentes

**Regla:** Usar funciones `input()` y `output()` en lugar de los decoradores `@Input()` y `@Output()`.

**Ejemplos:**
```typescript
// ✅ Correcto - usando input() y output()
import { Component, input, output } from '@angular/core';

@Component({
  selector: 'ft-user-card',
  template: `...`
})
export class UserCard {
  user = input.required<User>();
  onSelect = output<User>();
  
  handleClick() {
    this.onSelect.emit(this.user());
  }
}

// ❌ Incorrecto - usando decoradores
export class UserCard {
  @Input() user!: User;
  @Output() onSelect = new EventEmitter<User>();
}
```

### 6. Signals para Gestión de Estado

**Regla:** Usar signals para gestión de estado reactiva.

**Ejemplos:**
```typescript
// ✅ Correcto - usando signals
import { signal, computed } from '@angular/core';

export class SalesList {
  sales = signal<Sale[]>([]);
  totalSales = computed(() => 
    this.sales().reduce((sum, sale) => sum + sale.amount, 0)
  );
  
  loadSales() {
    this.sales.set([...]);
  }
}

// ❌ Incorrecto - usando propiedades de clase
export class SalesList {
  sales: Sale[] = [];
  totalSales = 0;
}
```

### 7. Flujo de Control en Templates

**Regla:** Usar flujo de control nativo (`@if`, `@for`, `@switch`) en lugar de directivas estructurales.

**Ejemplos:**
```html
<!-- ✅ Correcto - flujo de control nativo -->
@if (user()) {
  <div>Welcome, {{ user().name }}</div>
}

@for (sale of sales(); track sale.id) {
  <app-sale-item [sale]="sale" />
}

@switch (status()) {
  @case ('loading') {
    <app-spinner />
  }
  @case ('error') {
    <app-error [message]="error()" />
  }
  @default {
    <app-content />
  }
}

<!-- ❌ Incorrecto - directivas estructurales -->
<div *ngIf="user">Welcome, {{ user.name }}</div>
<div *ngFor="let sale of sales">
  <app-sale-item [sale]="sale" />
</div>
```

### 8. Bindings de Clase y Estilo

**Regla:** Preferir bindings `[class]` y `[style]` sobre `ngClass` y `ngStyle`.

**Ejemplos:**
```html
<!-- ✅ Correcto - bindings de clase y estilo -->
<div [class.active]="isActive()" [class.disabled]="isDisabled()">
<div [style.color]="textColor()" [style.font-size.px]="fontSize()">

<!-- ❌ Incorrecto - ngClass y ngStyle -->
<div [ngClass]="{ active: isActive(), disabled: isDisabled() }">
<div [ngStyle]="{ color: textColor(), fontSize: fontSize() + 'px' }">
```

### 9. Selectores de Componentes

**Regla:** Usar un prefijo de aplicación consistente para todos los selectores de componentes y directivas.

**Ejemplo:**
```typescript
// ✅ Correcto - usando prefijo 'ft'
@Component({
  selector: 'ft-user-card',
  // ...
})

@Component({
  selector: 'ft-sales-list',
  // ...
})

// ❌ Incorrecto - prefijos inconsistentes
@Component({
  selector: 'app-user-card',  // prefijo diferente
  // ...
})
```

### 10. Un Concepto Por Archivo

**Regla:** Mantener archivos enfocados en una sola responsabilidad. Una clase, interfaz o tipo por archivo.

**Ejemplos:**
```typescript
// ✅ Correcto - una clase por archivo
// user.model.ts
export interface User {
  id: string;
  name: string;
}

// user.service.ts
export class UserService {
  // ...
}

// ❌ Incorrecto - múltiples conceptos en un archivo
// user.ts
export interface User { }
export class UserService { }
export class UserView { }
```

### 11. Componentes Standalone

**Regla:** Siempre usar componentes standalone (por defecto en Angular moderno).

**Ejemplo:**
```typescript
// ✅ Correcto - componente standalone (por defecto)
@Component({
  selector: 'ft-user-card',
  standalone: true,  // o omitir, ya que es el defecto
  imports: [CommonModule, Button],
  template: `...`
})
export class UserCard {}

// ❌ Incorrecto - componente basado en NgModule
@NgModule({
  declarations: [UserCard],
  // ...
})
export class UserModule {}
```

## Implementación

Todas las reglas de la Guía de Estilo de Angular están documentadas en `.cursor/rules/` y se aplican automáticamente cuando se usa Cursor para desarrollo asistido por IA. Esto asegura:

- Generación de código consistente
- Aplicación automática durante el desarrollo
- Reducción de la necesidad de revisiones manuales de código en temas de estilo
- Onboarding más rápido para nuevos desarrolladores

## Consecuencias

### Positivas

- **Consistencia:** Todo el código sigue los mismos patrones y convenciones
- **Mantenibilidad:** Más fácil leer y entender código escrito por diferentes desarrolladores
- **Mejores Prácticas:** Aprovecha las recomendaciones del equipo de Angular para rendimiento óptimo
- **Onboarding:** Los nuevos desarrolladores pueden entender rápidamente el codebase
- **Herramientas:** Mejor soporte de IDE y autocompletado
- **Preparado para el futuro:** Alineado con la dirección y evolución de Angular

### Negativas

- **Curva de Aprendizaje:** Los miembros del equipo necesitan aprender y recordar las convenciones
- **Migración:** El código existente puede necesitar refactorización para coincidir con la guía
- **Rigidez:** Menos flexibilidad en el estilo de codificación (aunque esto es generalmente positivo)

### Mitigación

- Documentación exhaustiva en `.cursor/rules/`
- Proceso de revisión de código para detectar desviaciones
- Linting y formateo automatizado (ESLint, Prettier)
- Sesiones de entrenamiento regulares del equipo
- Herramientas de desarrollo asistido por IA para hacer cumplir las reglas automáticamente
