# ADR-006: Repository Pattern for REST Services and Local Async Persistence

**📖 [English](./ADR-006.md) | [Español](../../es/adr/ADR-006.md)**

**Status:** Accepted  
**Date:** 2025  
**Deciders:** Architecture Team

## Context

As the application grows, managing asynchronous persistence (REST services, IndexedDB, and similar storage layers) becomes complex. Most of our day-to-day examples use REST services, so the patterns below illustrate HTTP scenarios, but they apply equally to IndexedDB-based repositories or any other async storage mechanism:
- Multiple services making HTTP requests with different patterns
- Inconsistent error handling across components
- Duplicated loading state management
- Lack of centralized API endpoint management
- Difficult to test and mock API calls
- No standardized way to handle mutations vs. queries

Without a consistent pattern, developers might:
- Create ad-hoc data calls in components
- Duplicate error handling logic
- Manage loading states manually in each component
- Make it difficult to test and maintain API interactions

## Decision

We will use the **Repository Pattern** with helper functions `getMutations()` and `getResource()` (defined in `@/core/utils/async-repository`) to standardize all asynchronous persistence operations. This pattern provides:

1. **Centralized API logic** in repository classes
2. **Automatic state management** with signals (loading, error, value)
3. **Consistent error handling** with automatic user notifications
4. **Type-safe API calls** with TypeScript generics
5. **Easy testing** through dependency injection
6. **Separation of concerns** between data access and business logic

## Implementation

### Repository Structure

Each entity has its own repository class that encapsulates all API operations:

```typescript
import { HttpClient } from '@angular/common/http';
import { inject, Injectable } from '@angular/core';
import { getMutations, getResource, getApiUrl, SignalGet } from '@/core/utils/async-repository';
import { Customer, CustomerRequest } from '@/features/customers/models/customer';

@Injectable({ providedIn: 'root' })
export class CustomerRepository {
  private readonly httpClient = inject(HttpClient);
  private readonly baseUrl = getApiUrl('v1/customers');

  // Mutations (POST, PUT, DELETE)
  public mutations() {
    return getMutations({
      create: (customer: CustomerRequest) =>
        this.httpClient.post<Customer>(this.baseUrl, customer),
      update: (customer: CustomerRequest) =>
        this.httpClient.put<Customer>(`${this.baseUrl}/${customer.id}`, customer),
      delete: (id: string) =>
        this.httpClient.delete<void>(`${this.baseUrl}/${id}`)
    });
  }

  // Single resource (GET by ID)
  public find(): SignalGet<string, Customer> {
    return getResource<string, Customer>((id: string) => {
      return this.httpClient.get<Customer>(`${this.baseUrl}/${id}`);
    });
  }

  // Resource list (GET all or filtered)
  public findBy(): SignalGet<void, Customer[]> {
    return getResource<void, Customer[]>(() => {
      return this.httpClient.get<Customer[]>(this.baseUrl);
    });
  }
}
```

### API Endpoint Resolution

Always use `getApiUrl()` to resolve API endpoints:

```typescript
// ✅ Correct
private readonly baseUrl = getApiUrl('v1/customers');

// ❌ Incorrect - hardcoded URL
private readonly baseUrl = 'https://api.example.com/v1/customers';
```

### Mutations (Write Operations)

Use `getMutations()` for POST, PUT, PATCH, and DELETE operations:

```typescript
// In component or service
const mutations = customerRepository.mutations();

// Create
mutations.create({ name: 'John', email: 'john@example.com' })
  .then(result => {
    console.log('Created:', result);
  })
  .catch(error => {
    // Error is automatically shown to user via MessageService
    // But you can still handle it if needed
  });

// Update
mutations.update({ id: '123', name: 'John Updated' })
  .then(result => console.log('Updated:', result));

// Delete
mutations.delete('123')
  .then(() => console.log('Deleted'));

// Access state signals
mutations.submitting();  // true if any mutation is running
mutations.error();       // error message if any mutation failed
mutations.create.submitting();  // true if create is running
mutations.create.value();       // last created value
mutations.create.error();       // create-specific error
```

### Resources (Read Operations)

Use `getResource()` for GET operations:

```typescript
// Single resource
const customerResource = customerRepository.find();

// Load data
customerResource.load('123').then(customer => {
  console.log('Customer:', customer);
});

// Access reactive signals
customerResource.value();   // current customer value
customerResource.loading(); // true while loading
customerResource.error();   // error message if failed

// Resource list
const customers = customerRepository.findBy();
customers.load().then(list => console.log('Customers:', list));

// Cleanup (important for memory management)
customerResource.destroy();
customers.destroy();
```

### Component Usage Example

```typescript
@Component({
  selector: 'ft-customer-list',
  template: `
    @if (customers.loading()) {
      <app-spinner />
    } @else if (customers.error()) {
      <app-error [message]="customers.error()" />
    } @else {
      @for (customer of customers.value(); track customer.id) {
        <app-customer-card [customer]="customer" />
      }
    }
    
    <button 
      [disabled]="mutations.submitting()"
      (click)="handleCreate()"
    >
      Create Customer
    </button>
  `
})
export class CustomerListComponent implements OnInit, OnDestroy {
  private customerRepo = inject(CustomerRepository);
  
  customers = this.customerRepo.findBy();
  mutations = this.customerRepo.mutations();

  ngOnInit() {
    this.customers.load();
  }

  ngOnDestroy() {
    this.customers.destroy();
  }

  async handleCreate() {
    try {
      const newCustomer = await this.mutations.create({
        name: 'New Customer',
        email: 'new@example.com'
      });
      // Reload list after creation
      this.customers.load();
    } catch (error) {
      // Error already shown to user, but can handle here if needed
    }
  }
}
```

## Key Features

### 1. Automatic State Management

Both `getMutations()` and `getResource()` automatically manage:
- **Loading/Submitting state**: Signals that indicate when operations are in progress
- **Error state**: Signals that contain error messages
- **Value state**: Signals that contain the result of operations

### 2. Automatic Error Handling

- Errors are automatically caught and displayed to users via `MessageService`
- Errors are still thrown, allowing try/catch blocks for custom handling
- Error messages are extracted from API responses in a consistent way

### 3. Type Safety

- Full TypeScript support with generics
- Type inference for request/response types
- Compile-time checking of API contracts

### 4. Memory Management

- Resources provide `destroy()` method to clean up subscriptions
- Always call `destroy()` in component `ngOnDestroy()` lifecycle hook
- Prevents memory leaks from lingering subscriptions

## Repository Naming Conventions

- **Repository class**: `{Entity}Repository` (e.g., `CustomerRepository`, `OrderRepository`)
- **Mutations method**: Always named `mutations()`
- **Single resource method**: Named `find()` or `findById()`
- **List resource method**: Named `findBy()` or `findAll()`
- **Filtered queries**: Named `findBy{Filter}()` (e.g., `findByStatus()`, `findByDateRange()`)

## Best Practices

### 1. One Repository Per Entity

```typescript
// ✅ Correct - one repository per entity
CustomerRepository
OrderRepository
ProductRepository

// ❌ Incorrect - multiple entities in one repository
DataRepository  // Too generic
```

### 2. Always Use getApiUrl()

```typescript
// ✅ Correct
private readonly baseUrl = getApiUrl('v1/customers');

// ❌ Incorrect
private readonly baseUrl = `${environment.apiUrl}/v1/customers`;
```

### 3. Always Destroy Resources

```typescript
// ✅ Correct
ngOnDestroy() {
  this.customers.destroy();
}

// ❌ Incorrect - memory leak
// Forgot to call destroy()
```

### 4. Use Signals for UI Binding

```typescript
// ✅ Correct - reactive signals
@if (customers.loading()) { ... }
@if (customers.error()) { ... }

// ❌ Incorrect - manual state management
loading = false;
error = null;
```

### 5. Handle Errors Appropriately

```typescript
// ✅ Correct - errors are auto-shown, but can still handle
try {
  await mutations.create(data);
  // Success handling
} catch (error) {
  // Custom error handling if needed
  // Error already shown to user
}

// ❌ Incorrect - ignoring errors
mutations.create(data);  // No error handling
```

## Consequences

### Positive

- **Consistency:** All API calls follow the same pattern
- **Maintainability:** Centralized API logic is easier to update
- **Testability:** Repositories can be easily mocked in tests
- **Type Safety:** Full TypeScript support prevents runtime errors
- **Developer Experience:** Less boilerplate code, automatic state management
- **Error Handling:** Consistent error handling across the application
- **Performance:** Automatic cleanup prevents memory leaks

### Negative

- **Learning Curve:** Team members need to learn the pattern
- **Abstraction:** Additional layer of abstraction (though beneficial)
- **Initial Setup:** Requires creating repository classes for each entity

### Mitigation

- Comprehensive documentation in this ADR
- Code examples and templates
- Code review process to ensure pattern compliance
- Clear naming conventions

## Testing

Repositories can be easily tested by mocking `HttpClient`:

```typescript
describe('CustomerRepository', () => {
  let repository: CustomerRepository;
  let httpMock: Partial<HttpClient>;

  beforeEach(() => {
    httpMock = createHttpClientMock();
    repository = new CustomerRepository(httpMock);
  });

  it('should load customer by id', async () => {
    const customer = { id: '123', name: 'John' };
    httpMock.get.mockReturnValue(of(customer));

    const resource = repository.find();
    const result = await resource.load('123');

    expect(result).toEqual(customer);
    expect(resource.value()).toBe(customer);
  });
});
```
