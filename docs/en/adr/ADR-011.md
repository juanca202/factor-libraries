# ADR-011: Documentation Strategy

**📖 [English](./ADR-011.md) | [Español](../../es/adr/ADR-011.md)**

**Status:** Accepted  
**Date:** 2025  
**Deciders:** Architecture Team

## Context

Clear and comprehensive code documentation is essential for:
- Onboarding new developers
- Maintaining code over time
- Understanding complex logic and business rules
- Generating API documentation automatically
- Ensuring consistency across the codebase

Without a clear documentation strategy:
- Documentation may be inconsistent or missing
- Comments might be in different languages, causing confusion
- API documentation requires manual maintenance
- Code reviews become harder without clear documentation standards
- New team members struggle to understand the codebase

We need a standardized approach that:
- Ensures all documentation is in English for consistency
- Automatically generates API documentation from code comments
- Uses a standard format (JSDoc/TSDoc) that tools can parse
- Integrates seamlessly into the development workflow
- Provides clear guidelines for when and how to document code

## Decision

We will implement a **comprehensive documentation strategy** with:

1. **English-only documentation:** All code comments, JSDoc/TSDoc annotations, and inline documentation must be written in English
2. **Compodoc:** Use Compodoc to automatically generate API documentation from JSDoc/TSDoc comments
3. **JSDoc/TSDoc format:** All public APIs, classes, methods, and complex logic must be documented using JSDoc or TSDoc annotations
4. **Documentation standards:** Clear guidelines on what to document and how

## Implementation

### 1. Language Requirement

**All documentation must be in English**, including:
- Code comments
- JSDoc/TSDoc annotations
- Inline documentation
- Function and method descriptions
- Parameter descriptions
- Return value descriptions
- Example code in documentation

**Rationale:**
- Ensures consistency across the codebase
- Makes the codebase accessible to international developers
- Aligns with industry best practices
- Simplifies tooling and automation

### 2. Compodoc Configuration

Compodoc is configured to automatically generate API documentation from JSDoc/TSDoc comments.

**Configuration:** `compodoc.json` or `package.json` scripts

**Usage:**
```bash
npm run compodoc        # Generate documentation
npm run compodoc:serve  # Generate and serve documentation locally
```

**Output:** Documentation is generated in the `documentation/` directory

### 3. JSDoc/TSDoc Format

Use **TSDoc** (TypeScript-specific) for TypeScript files and **JSDoc** for JavaScript files.

#### TSDoc Format (Recommended for TypeScript)

TSDoc is the recommended format for TypeScript projects as it provides better type awareness and integration with TypeScript tooling.

**Basic Example:**
```typescript
/**
 * Calculates the total price including tax.
 *
 * @param basePrice - The base price before tax
 * @param taxRate - The tax rate as a decimal (e.g., 0.10 for 10%)
 * @returns The total price including tax
 *
 * @example
 * ```typescript
 * const total = calculateTotalPrice(100, 0.10);
 * console.log(total); // 110
 * ```
 */
export function calculateTotalPrice(basePrice: number, taxRate: number): number {
  return basePrice * (1 + taxRate);
}
```

**Class Example:**
```typescript

   */

```

**Component Example:**
```typescript
/**
 * User profile component that displays and allows editing of user information.
 *
 * @component
 * @selector app-user-profile
 *
 * @example
 * ```html
 * <app-user-profile [userId]="123"></app-user-profile>

### 4. Documentation Standards

#### What to Document

**Must Document:**
- All public APIs (exported classes, functions, interfaces, types)
- Complex algorithms or business logic
- Non-obvious code behavior
- Public component inputs and outputs
- Service methods and their parameters
- Custom validators and pipes
- Utility functions with non-trivial logic

**Should Document:**
- Private methods with complex logic
- Workarounds or temporary solutions
- Performance considerations
- Known limitations or edge cases

**Optional:**
- Simple getters/setters
- Self-explanatory code
- Standard Angular lifecycle hooks (unless custom behavior)

#### Documentation Tags

**Common TSDoc/JSDoc Tags:**

- `@param` - Parameter description
- `@returns` or `@return` - Return value description
- `@throws` or `@throws` - Exceptions that may be thrown
- `@example` - Usage examples
- `@remarks` - Additional notes
- `@deprecated` - Mark deprecated APIs
- `@since` - Version when the API was introduced
- `@see` - Reference to related documentation
- `@internal` - Mark internal APIs (not included in public docs)

**Example with Multiple Tags:**
```typescript
/**
 * Validates an email address format.
 *
 * @param email - The email address to validate
 * @returns True if the email format is valid, false otherwise
 * @throws {TypeError} If email is not a string
 *
 * @example
 * ```typescript
 * isValidEmail('user@example.com'); // true
 * isValidEmail('invalid-email');     // false
 * ```
 *
 * @remarks
 * This validator checks format only, not domain existence.
 * For production use, consider additional validation.
 *
 * @since 1.0.0
 */
export function isValidEmail(email: string): boolean {
  if (typeof email !== 'string') {
    throw new TypeError('Email must be a string');
  }
  const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  return emailRegex.test(email);
}
```

### 5. Compodoc Integration

Compodoc automatically scans the codebase for JSDoc/TSDoc comments and generates:
- API documentation
- Component documentation
- Service documentation
- Module documentation
- Route documentation

**Configuration Example:**
```json
{
  "name": "angular-base-project",
  "compodoc": {
    "output": "documentation",
    "theme": "gitbook",
    "includes": "src/**/*.ts",
    "includesName": "API Documentation"
  }
}
```

**Generated Documentation Includes:**
- Class and interface documentation
- Method and property descriptions
- Parameter and return type information
- Usage examples
- Module dependency graphs
- Component tree visualization

## Consequences

### Positive

- **Consistency:** All documentation in English ensures uniform understanding
- **Automation:** Compodoc automatically generates API documentation
- **Maintainability:** Standard format makes documentation easy to update
- **Developer Experience:** Clear documentation helps onboarding and development
- **Type Safety:** TSDoc provides better integration with TypeScript
- **Tooling Support:** JSDoc/TSDoc is widely supported by IDEs and tools
- **Professional:** Well-documented codebase reflects professional standards

### Negative

- **Initial Overhead:** Requires time to document existing code
- **Maintenance:** Documentation must be kept in sync with code changes
- **Learning Curve:** Team members need to learn JSDoc/TSDoc conventions
- **English Requirement:** May be challenging for non-native English speakers

### Mitigation

- **Gradual Adoption:** Document new code first, gradually update existing code
- **Code Reviews:** Include documentation review in pull request process
- **Templates:** Provide JSDoc/TSDoc templates for common patterns
- **Training:** Offer documentation best practices training
- **Automation:** Use linting rules to enforce documentation on public APIs
- **Examples:** Maintain a library of well-documented code examples

## Best Practices

1. **Write documentation as you code:** Don't leave it for later
2. **Keep it concise:** Focus on what, why, and how, not obvious details
3. **Use examples:** Code examples clarify usage better than descriptions
4. **Update with code:** When changing code, update documentation too
5. **Review documentation:** Include documentation in code reviews
6. **Use meaningful names:** Good naming reduces need for extensive documentation
7. **Document the "why":** Explain reasoning, not just what the code does
