# ADR-010: Icon Usage Strategy

**📖 [English](./ADR-010.md) | [Español](../../es/adr/ADR-010.md)**

**Status:** Accepted  
**Date:** 2025  
**Deciders:** Architecture Team

## Context

Icons are a fundamental part of the user interface, and without a standardized approach:
- Different developers may use different icon libraries or approaches
- Inconsistent icon rendering across the application
- Difficult to maintain and update icons
- No clear pattern for custom icons
- Potential performance issues from multiple icon libraries
- Accessibility concerns with inconsistent icon implementations

We need a consistent strategy that:
- Standardizes icon usage across the application
- Provides a single component for all icon rendering
- Supports both predefined and custom icons
- Ensures accessibility and performance
- Maintains visual consistency

## Decision

We will use a **unified icon component** (`<ft-icon />`) that supports multiple icon collections and custom icons. The implementation includes:

1. **Single Icon Component:** `<ft-icon />` for all icon rendering
2. **Predefined Collections:** factoricons-slim, factoricons-regular, factoricons-solid
3. **Custom Icon Library:** Custom icons stored in `public/images/icons.svg`
4. **Size Modifiers:** Standardized size classes (`ft-icon--1` to `ft-icon--5`)
5. **Material Design Alignment:** All icons align with Material 24×24 grid

## Implementation

### Icon Component Usage

Always use the `<ft-icon />` component for all icons in the application:

```html
<!-- ✅ Correct - using ft-icon component -->
<ft-icon name="user" />
<ft-icon name="settings" collection="factoricons-slim" />
<ft-icon name="custom-icon" collection="icons" />

<!-- ❌ Incorrect - using other icon approaches -->
<mat-icon>person</mat-icon>
<svg>...</svg>
<i class="material-icons">person</i>
```

### Available Collections

#### Predefined Collections

- **`factoricons-regular`** (default) - No need to specify the property
- **`factoricons-slim`** - Slim variant
- **`factoricons-solid`** - Solid variant

#### Custom Icon Collection

- **`icons`** - Custom icons stored in `public/images/icons.svg`

### Component Properties

The `<ft-icon />` component accepts:

- **`name`** (required): The icon's symbol name
- **`collection`** (optional): The icon collection source (defaults to `factoricons-regular`)

### Custom Icon Library

When using the `icons` collection, icons must exist in `public/images/icons.svg`.

#### Creating Custom Icons

Each custom icon should be defined as an **SVG symbol** optimized to **24×24 pixels**:

```html
<!-- public/images/icons.svg -->
<svg xmlns="http://www.w3.org/2000/svg" style="display: none;">
  <symbol id="settings" viewBox="0 0 24 24">
    <path fill="currentColor" d="M12 8a4 4 0 1 0 0 8a4 4 0 0 0 0-8z" />
  </symbol>
  
  <symbol id="user-settings" viewBox="0 0 24 24">
    <path fill="currentColor" d="M12 12a4 4 0 1 0 0-8a4 4 0 0 0 0 8z" />
  </symbol>
</svg>
```

**Requirements:**
- Each icon must be a `<symbol>` element
- `id` attribute acts as the icon name (used in `name` property)
- `viewBox` should be `0 0 24 24` for Material Design alignment
- Use `fill="currentColor"` to inherit text color
- Optimize paths for clarity and performance

### Icon Sizes

To adjust icon size, use the **size modifier class** `ft-icon--{n}` where `{n}` ranges from **1 to 5**:

```html
<!-- ✅ Correct - using size modifiers -->
<ft-icon name="user" class="ft-icon--1" />  <!-- Extra small -->
<ft-icon name="user" class="ft-icon--2" />  <!-- Small -->
<ft-icon name="user" class="ft-icon--3" />  <!-- Medium (default) -->
<ft-icon name="user" class="ft-icon--4" />  <!-- Large -->
<ft-icon name="user" class="ft-icon--5" />  <!-- Extra large -->
```

**Size Scale:**
- `ft-icon--1`: Extra small
- `ft-icon--2`: Small
- `ft-icon--3`: Medium (default)
- `ft-icon--4`: Large
- `ft-icon--5`: Extra large

**Custom Sizes:**

If a specific size is required beyond the predefined scale, apply a custom size using `font-size`:

```html
<!-- ✅ Correct - custom size -->
<ft-icon name="user" style="font-size: 32px;" />
```

### Usage Examples

#### Using Default Collection (factoricons-regular)

```html
<ft-icon name="user" />
<ft-icon name="settings" />
<ft-icon name="home" />
```

#### Using Another Predefined Collection

```html
<ft-icon name="user" collection="factoricons-slim" />
<ft-icon name="settings" collection="factoricons-solid" />
```

#### Using Custom Icons

```html
<ft-icon name="user-settings" collection="icons" />
<ft-icon name="calendar-add" collection="icons" />
```

#### With Size Modifiers

```html
<ft-icon name="user" class="ft-icon--2" />
<ft-icon name="settings" class="ft-icon--4" />
```

#### In Components

```html
<button mat-flat-button color="primary">
  <ft-icon name="save" class="ft-icon--2" />
  <span i18n>Save</span>
</button>

<mat-form-field appearance="outline">
  <mat-label i18n>Search</mat-label>
  <ft-icon name="search" matPrefix class="ft-icon--2" />
  <input matInput type="text" />
</mat-form-field>
```

## Best Practices

### 1. Prefer Predefined Collections

```html
<!-- ✅ Correct - use predefined collections when possible -->
<ft-icon name="user" />
<ft-icon name="settings" collection="factoricons-slim" />

<!-- ❌ Incorrect - creating custom icon when predefined exists -->
<ft-icon name="user-icon" collection="icons" />
```

### 2. Only Add Custom Icons When Necessary

- Only add new icons to the `icons` collection when no suitable symbol exists in the predefined sets
- Search within the desired collection first before creating custom icons
- Custom icons should follow Material Design 24×24 grid guidelines

### 3. Use Size Modifiers Consistently

```html
<!-- ✅ Correct - use standard size modifiers -->
<ft-icon name="user" class="ft-icon--3" />

<!-- ❌ Incorrect - inline styles for standard sizes -->
<ft-icon name="user" style="font-size: 24px;" />
```

### 4. Maintain Visual Consistency

- All icons (custom or predefined) must visually align with the **Material 24×24 grid** for consistency
- Use `fill="currentColor"` in custom SVG symbols to inherit text color
- Ensure icons are optimized and clear at all sizes

### 5. Icon Selection Guidelines

- To select an icon, **search within the desired collection** for a symbol whose `id` best represents the intended meaning or concept
- The `id` of each `<symbol>` acts as the **descriptive identifier** and is used in the `name` property of the `<ft-icon />` component
- Choose icon names that are clear and descriptive

## Icon Naming Conventions

### Predefined Collections

- Use kebab-case for icon names
- Names should be descriptive and clear
- Examples: `user`, `settings`, `home`, `arrow-right`, `check-circle`

### Custom Icons

- Use kebab-case for custom icon IDs
- Use descriptive compound names when needed
- Examples: `user-settings`, `calendar-add`, `file-upload`

## Accessibility

- Icons should be accompanied by text labels when used as interactive elements
- Use `aria-label` or `aria-labelledby` for icon-only buttons
- Ensure sufficient color contrast for icons
- Icons should be recognizable and clear at all sizes

## Consequences

### Positive

- **Consistency:** All icons use the same component and approach
- **Maintainability:** Single source of truth for icon rendering
- **Performance:** Optimized icon loading and rendering
- **Flexibility:** Support for both predefined and custom icons
- **Accessibility:** Consistent accessibility patterns
- **Visual Consistency:** All icons align with Material Design grid

### Negative

- **Learning Curve:** Team members need to learn the icon component API
- **Custom Icon Creation:** Requires SVG knowledge for custom icons
- **File Management:** Custom icons require maintaining `icons.svg` file

### Mitigation

- Clear documentation and examples
- Code review process to ensure proper usage
- Icon library search tools and documentation
- Regular review of custom icons to ensure they're still needed
