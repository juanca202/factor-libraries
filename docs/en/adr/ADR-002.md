# ADR-002: Adoption of the Official Angular Style Guide

**📖 [English](./ADR-002.md) | [Español](../../es/adr/ADR-002.md)**

**Status:** Accepted  
**Date:** 2025  
**Deciders:** Architecture Team

## Context

As Angular evolves and the team grows, maintaining code consistency becomes challenging. Different developers may have different coding styles, leading to:
- Inconsistent code patterns across the codebase
- Difficulty in code reviews
- Slower onboarding for new developers
- Maintenance challenges
- Potential performance issues from non-optimal patterns

The Angular team provides an official [Angular Style Guide](https://angular.dev/style-guide) that represents best practices and recommended patterns for Angular applications.

## Decision

We will adopt and strictly follow the official Angular Style Guide to ensure consistency, maintainability, and alignment with Angular best practices.

All code in this project must conform to the Angular Style Guide, and these rules are documented in `.cursor/rules/` to be automatically enforced by AI-assisted development tools.

## Key Aspects Applied

### 1. File Naming Conventions

**Rule:** Use kebab-case (hyphens) to separate words in file names. Required suffixes must also use hyphens (e.g., `-guard`, `-validator`).

**Required suffixes:** Only Guards, Pipes, Resolvers, Interceptors, and Validators must include their suffix in both the class name and the file name (for example, `AuthGuard` lives in `auth-guard.ts`). All other artifacts (components, services, models, utilities, etc.) must avoid artificial suffixes such as `-component` or `-model`.

**Examples:**
```typescript
// ✅ Correct
user-profile.ts
sales-list.ts
http-error-interceptor.ts
email-validator.ts

// ❌ Incorrect
userProfile.ts
salesList.ts
httpError-interceptor.ts
emailValidator.ts
```

### 2. Class Naming Conventions

**Rule:** Use PascalCase for classes, interfaces, and types.

**Suffix standard:** Guard, Pipe, Resolver, Interceptor, and Validator classes must end with their respective suffix and the file name must mirror it (e.g., `auth-guard.ts`, `format-currency-pipe.ts`).

**Examples:**
```typescript
// ✅ Correct
export class UserProfile {}
export interface UserModel {}
export type UserRole = 'admin' | 'user';
export class AuthGuard {}
export class FormatCurrencyPipe {}
export class LoadCustomerResolver {}
export class HttpErrorInterceptor {}

// ❌ Incorrect
export class userProfile {}
export interface userModel {}
export class Auth {}
export class FormatCurrency {}
export class LoadCustomer {}
export class HttpError {}
```

### 3. Project Structure

**Rule:** Organize code by feature, not by file type.

**Example Structure:**
```
// ✅ Correct - organized by feature
sales/
  ├── components/
  ├── services/
  ├── models/
  └── sales-routes.ts

// ❌ Incorrect - organized by type
components/
  ├── sales-list.ts
  ├── sales-detail.ts
services/
  ├── sales-service.ts
models/
  └── sale.ts
```

### 4. Dependency Injection

**Rule:** Prefer the `inject()` function over constructor injection.

**Examples:**
```typescript
// ✅ Correct - using inject()
import { inject } from '@angular/core';
import { HttpClient } from '@angular/common/http';

export class SalesService {
  private http = inject(HttpClient);
  private logger = inject(Logger);
}

// ❌ Incorrect - constructor injection
export class SalesService {
  constructor(
    private http: HttpClient,
    private logger: Logger
  ) {}
}
```

### 5. Component Inputs and Outputs

**Rule:** Use `input()` and `output()` functions instead of `@Input()` and `@Output()` decorators.

**Examples:**
```typescript
// ✅ Correct - using input() and output()
import { Component, input, output } from '@angular/core';

@Component({
  selector: 'ft-user-card',
  template: `...`
})
export class UserCard {
  user = input.required<User>();
  onSelect = output<User>();
  
  handleClick() {
    this.onSelect.emit(this.user());
  }
}

// ❌ Incorrect - using decorators
export class UserCard {
  @Input() user!: User;
  @Output() onSelect = new EventEmitter<User>();
}
```

### 6. Signals for State Management

**Rule:** Use signals for reactive state management.

**Examples:**
```typescript
// ✅ Correct - using signals
import { signal, computed } from '@angular/core';

export class SalesList {
  sales = signal<Sale[]>([]);
  totalSales = computed(() => 
    this.sales().reduce((sum, sale) => sum + sale.amount, 0)
  );
  
  loadSales() {
    this.sales.set([...]);
  }
}

// ❌ Incorrect - using class properties
export class SalesList {
  sales: Sale[] = [];
  totalSales = 0;
}
```

### 7. Template Control Flow

**Rule:** Use native control flow (`@if`, `@for`, `@switch`) instead of structural directives.

**Examples:**
```html
<!-- ✅ Correct - native control flow -->
@if (user()) {
  <div>Welcome, {{ user().name }}</div>
}

@for (sale of sales(); track sale.id) {
  <app-sale-item [sale]="sale" />
}

@switch (status()) {
  @case ('loading') {
    <app-spinner />
  }
  @case ('error') {
    <app-error [message]="error()" />
  }
  @default {
    <app-content />
  }
}

<!-- ❌ Incorrect - structural directives -->
<div *ngIf="user">Welcome, {{ user.name }}</div>
<div *ngFor="let sale of sales">
  <app-sale-item [sale]="sale" />
</div>
```

### 8. Class and Style Bindings

**Rule:** Prefer `[class]` and `[style]` bindings over `ngClass` and `ngStyle`.

**Examples:**
```html
<!-- ✅ Correct - class and style bindings -->
<div [class.active]="isActive()" [class.disabled]="isDisabled()">
<div [style.color]="textColor()" [style.font-size.px]="fontSize()">

<!-- ❌ Incorrect - ngClass and ngStyle -->
<div [ngClass]="{ active: isActive(), disabled: isDisabled() }">
<div [ngStyle]="{ color: textColor(), fontSize: fontSize() + 'px' }">
```

### 9. Component Selectors

**Rule:** Use a consistent application prefix for all component and directive selectors.

**Example:**
```typescript
// ✅ Correct - using 'ft' prefix
@Component({
  selector: 'ft-user-card',
  // ...
})

@Component({
  selector: 'ft-sales-list',
  // ...
})

// ❌ Incorrect - inconsistent prefixes
@Component({
  selector: 'app-user-card',  // different prefix
  // ...
})
```

### 10. One Concept Per File

**Rule:** Keep files focused on a single responsibility. One class, interface, or type per file.

**Examples:**
```typescript
// ✅ Correct - one class per file
// user.model.ts
export interface User {
  id: string;
  name: string;
}

// user.service.ts
export class UserService {
  // ...
}

// ❌ Incorrect - multiple concepts in one file
// user.ts
export interface User { }
export class UserService { }
export class UserView { }
```

### 11. Standalone Components

**Rule:** Always use standalone components (default in modern Angular).

**Example:**
```typescript
// ✅ Correct - standalone component (default)
@Component({
  selector: 'ft-user-card',
  standalone: true,  // or omit, as it's the default
  imports: [CommonModule, Button],
  template: `...`
})
export class UserCard {}

// ❌ Incorrect - NgModule-based component
@NgModule({
  declarations: [UserCard],
  // ...
})
export class UserModule {}
```

## Implementation

All Angular Style Guide rules are documented in `.cursor/rules/` and are automatically applied when using Cursor for AI-assisted development. This ensures:

- Consistent code generation
- Automatic enforcement during development
- Reduced need for manual code reviews on style issues
- Faster onboarding for new developers

## Consequences

### Positive

- **Consistency:** All code follows the same patterns and conventions
- **Maintainability:** Easier to read and understand code written by different developers
- **Best Practices:** Leverages Angular team's recommendations for optimal performance
- **Onboarding:** New developers can quickly understand the codebase
- **Tooling:** Better IDE support and autocomplete
- **Future-proof:** Aligned with Angular's direction and evolution

### Negative

- **Learning Curve:** Team members need to learn and remember the conventions
- **Migration:** Existing code may need refactoring to match the guide
- **Rigidity:** Less flexibility in coding style (though this is generally positive)

### Mitigation

- Comprehensive documentation in `.cursor/rules/`
- Code review process to catch deviations
- Automated linting and formatting (ESLint, Prettier)
- Regular team training sessions
- AI-assisted development tools to enforce rules automatically
