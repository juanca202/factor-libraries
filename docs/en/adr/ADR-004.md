# ADR-004: AI-Assisted Development Rules

**📖 [English](./ADR-004.md) | [Español](../../es/adr/ADR-004.md)**

**Status:** Accepted  
**Date:** 2025  
**Deciders:** Architecture Team

## Context

As AI-assisted development tools become more prevalent, there's a risk that:
- Generated code may not follow project conventions
- Architectural decisions may be overlooked or violated
- Code quality and consistency may degrade
- Different developers using AI tools may produce inconsistent code
- Best practices and patterns may not be consistently applied

Without clear rules and documentation, AI tools cannot effectively assist developers while maintaining the project's architectural integrity and code quality standards.

## Decision

We will establish comprehensive development rules documented in `docs/` that can be automatically loaded by AI-assisted development tools (e.g., through configuration files like `.cursor/rules.json`) to ensure:

1. Consistent code generation aligned with project architecture
2. Automatic enforcement of architectural decisions
3. Adherence to Angular Style Guide and project conventions
4. Quality and maintainability of AI-generated code

## Implementation

### Rule Documentation Structure

All architectural decisions and style rules are documented in `docs/` as Markdown files. AI-assisted development tools can automatically load these files through configuration files (e.g., `.cursor/rules.json` for Cursor):

```json
[
  {
    "name": "Base architecture docs",
    "description": "Load all architecture documentation for better suggestions.",
    "priority": "high",
    "match": ["**/*"],
    "paths": ["docs/**/*.md"]
  }
]
```

The documentation structure in `docs/` includes:

```
docs/
├── README.md                # Overall architecture and C4 diagrams
├── css-styling.md                 # CSS and Tailwind guidelines
└── adr/                          # Architectural Decision Records
    ├── ADR-001.md
    ├── ADR-002.md
    ├── ADR-003.md
    ├── ADR-004.md
    └── ...
```

**Note:** There is a distinction between:
- **Architecture documentation** (`docs/*.md`): Source of truth for architectural rules, can be loaded by AI-assisted development tools via configuration files
- **IDE-specific rules** (e.g., `.cursor/rules/*.mdc` for Cursor): Additional IDE-specific configuration and always-applied rules

## How Rules Are Applied

### Automatic Application via Configuration Files

AI-assisted development tools can automatically load all documentation from `docs/` through configuration files. For example, Cursor uses `.cursor/rules.json`:

```json
{
  "name": "Base architecture docs",
  "description": "Load all architecture documentation for better suggestions.",
  "priority": "high",
  "match": ["**/*"],
  "paths": ["docs/**/*.md"]
}
```

This configuration ensures that:

1. **Context Awareness:** The AI tool reads all Markdown files from `docs/` automatically
2. **Code Generation:** Generated code follows documented patterns and conventions from the architecture docs
3. **Suggestions:** AI suggestions align with architectural decisions documented in `docs/`
4. **Refactoring:** Refactoring suggestions maintain project structure as defined in the documentation
5. **Priority:** High priority ensures these rules are always considered during code generation

### Example: Code Generation

**Developer Request:**
> "Create a new sales list component"

**AI Response (following rules):**
```typescript
// ✅ Generated code follows all rules
@Component({
  selector: 'ft-sales-list',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  imports: [CommonModule, SaleItemComponent],
  template: `
    <div class="flex flex-col gap-4 p-6">
      <h2 class="text-2xl font-bold">Sales</h2>
      @for (sale of sales(); track sale.id) {
        <ft-sale-item [sale]="sale" />
      }
    </div>
  `
})
export class SalesListComponent {
  private salesService = inject(SalesService);
  
  sales = this.salesService.sales;
}
```

**Rules Applied:**
- ✅ Standalone component
- ✅ OnPush change detection
- ✅ `inject()` instead of constructor
- ✅ Signals for state
- ✅ Native control flow (`@for`)
- ✅ Tailwind utilities for styling
- ✅ Proper selector prefix (`ft-`)

- **Best Practices:** AI automatically applies Angular and project best practices

### 3. Onboarding

## Rule Maintenance

### Adding New Rules

1. **Identify Need:** Recognize a pattern or convention that should be standardized
2. **Document Rule:** Add rule to appropriate Markdown file in `docs/` (e.g., `docs/css-styling.md`) or create a new ADR in `docs/adr/` if it represents a significant architectural decision
3. **Provide Examples:** Include correct and incorrect examples in the documentation
4. **Update ADRs:** If rule represents a significant architectural decision, create or update an ADR in `docs/adrs/`
5. **Verify Loading:** Ensure the new documentation is automatically loaded by checking the IDE configuration file (e.g., `.cursor/rules.json` for Cursor) includes `docs/**/*.md`
6. **Team Review:** Share new rules with the team

### Updating Existing Rules

1. **Identify Change:** Recognize when a rule needs modification
2. **Update Documentation:** Modify the rule in the appropriate file in `docs/`
3. **Update Examples:** Ensure examples reflect the new rule
4. **Update ADRs:** If the change represents a significant decision, update the relevant ADR
5. **Communicate:** Notify team of rule changes
6. **Refactor Code:** Update existing code to match new rules if necessary

**Note:** Since `rules.json` uses the pattern `docs/**/*.md`, any new or updated Markdown files in `docs/` are automatically included without modifying `rules.json`.

## Architectural Decision Documentation Standard

All significant architectural decisions follow the **Architectural Decision Record (ADR)** format, which is the standard for documenting architectural decisions in this project.

### ADR Format

Each ADR follows a consistent structure:

- **Title:** Clear, descriptive title (e.g., `ADR-XXX: Decision Name`)
- **Status:** Current status (Accepted, Proposed, Deprecated, etc.)
- **Date:** When the decision was made
- **Deciders:** Who made the decision
- **Context:** The situation and problem that led to this decision
- **Decision:** The architectural choice that was made
- **Implementation:** Detailed implementation guidelines and examples
- **Consequences:** Positive and negative impacts of the decision
- **References:** Links to related documentation and resources

### ADR Location

All ADRs are located in `docs/en/adr/` (English) and `docs/es/adr/` (Spanish), following the naming convention `ADR-XXX.md` where XXX is a sequential number.

### Key Points

- **Standard Format:** All architectural decisions use the ADR format for consistency
- **Version Control:** ADRs are versioned with the codebase
- **Bilingual:** ADRs are maintained in both English and Spanish
- **AI Context:** ADRs serve as both human-readable documentation and context for AI-assisted development tools
- **Living Documentation:** ADRs are updated as decisions evolve

For more information about ADRs, see [Architectural Decision Records (ADRs)](./README.md).

## Consequences

### Positive

- **Consistency:** All AI-generated code follows project standards
- **Quality:** Automatic application of best practices
- **Speed:** Faster development with AI assistance
- **Onboarding:** New developers learn patterns through AI suggestions
- **Documentation:** Rules serve as executable documentation
- **Scalability:** Easy to add new rules as project evolves

### Negative

- **Initial Setup:** Requires time to document all rules comprehensively
- **Maintenance:** Rules need to be kept up-to-date with Angular and project evolution
- **Rigidity:** May limit flexibility in edge cases (though generally positive)
- **Tool Dependency:** Relies on AI tool's ability to read and apply rules

### Mitigation

- Start with essential rules and expand gradually
- Regular review and update of rules
- Balance between strictness and flexibility
- Document exceptions and edge cases
- Regular team feedback on rule effectiveness
