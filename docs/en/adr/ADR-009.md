# ADR-009: Code Quality & Tooling

**📖 [English](./ADR-009.md) | [Español](../../es/adr/ADR-009.md)**

**Status:** Accepted  
**Date:** 2025  
**Deciders:** Architecture Team

## Context

Maintaining code quality and consistency across a team requires automated tooling and clear standards. Without proper tooling:

- Code style inconsistencies across developers
- Bugs and issues discovered late in development
- Difficult code reviews focusing on style instead of logic
- Inconsistent commit messages making history hard to navigate
- Manual formatting and linting is time-consuming and error-prone
- No automated enforcement of coding standards

We need a comprehensive set of tools that:
- Enforces consistent code style automatically
- Catches errors and potential bugs early
- Validates commit messages for better project history
- Integrates seamlessly into the development workflow
- Requires minimal manual intervention

## Decision

We will use a **comprehensive code quality toolchain** with:

1. **ESLint:** Code linting and error detection
2. **Prettier:** Automatic code formatting
3. **Husky:** Git hooks for pre-commit and commit-msg validation
4. **lint-staged:** Run linters on staged files only
5. **Commitlint:** Enforce Conventional Commits format
6. **TypeScript:** Strict type checking
7. **Compodoc:** Automatic API documentation generation

All tools are configured to work together and enforce standards automatically.

## Implementation

### Tool Stack

#### 1. ESLint

**Purpose:** Static code analysis and linting

**Configuration:** `eslint.config.js`

**Key Features:**
- TypeScript-specific rules
- Angular-specific rules via `@angular-eslint`
- Template linting for Angular templates
- Integration with Prettier

**Usage:**
```bash
# Check for linting errors
npm run lint

# Auto-fix linting errors
npm run lint:fix
```

**Key Rules:**
- TypeScript strict mode enforcement
- Angular best practices
- Unused variables detection
- Import ordering
- Code complexity limits

#### 2. Prettier

**Purpose:** Automatic code formatting

**Configuration:** `.prettierrc`

**Settings:**
```json
{
  "printWidth": 100,
  "singleQuote": true,
  "trailingComma": "none",
  "overrides": [
    {
      "files": "*.html",
      "options": {
        "parser": "angular"
      }
    }
  ]
}
```

**Usage:**
```bash
# Format all files
npm run prettier
```

**Formats:**
- TypeScript/JavaScript files
- HTML templates
- SCSS/CSS files
- JSON files
- Markdown files

#### 3. Husky

**Purpose:** Git hooks for automated checks

**Hooks Configured:**

**pre-commit** (`.husky/pre-commit`):
```bash
npx lint-staged --no-stash
```
- Runs linting and formatting on staged files
- Prevents commits with linting errors
- Auto-fixes issues when possible

**commit-msg** (`.husky/commit-msg`):
```bash
npx --no -- commitlint --edit "$1"
```
- Validates commit message format
- Enforces Conventional Commits standard

#### 4. lint-staged

**Purpose:** Run linters only on staged files

**Configuration:** `package.json`
```json
{
  "lint-staged": {
    "src/**/*.{ts,js,html,scss,json,md}": [
      "prettier --write",
      "eslint --fix"
    ]
  }
}
```

**Benefits:**
- Faster pre-commit checks
- Only processes changed files
- Auto-fixes issues before commit

#### 5. Commitlint

**Purpose:** Enforce commit message conventions

**Configuration:** `commitlint.config.js`
```javascript
module.exports = {
  extends: ['@commitlint/config-conventional']
};
```

**Conventional Commits Format:**
```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

**Allowed Types:**
- `feat`: New feature
- `fix`: Bug fix
- `docs`: Documentation changes
- `style`: Code style changes (formatting, etc.)
- `refactor`: Code refactoring
- `test`: Adding or updating tests
- `chore`: Maintenance tasks

**Examples:**
```bash
# ✅ Valid commits
feat(auth): add Google login
fix(api): correct error handling in user endpoint
docs(readme): update installation instructions
style(components): apply prettier formatting
refactor(core): optimize notification service
test(auth): add login flow tests
chore(deps): update Angular to v20

# ❌ Invalid commits
Added new feature
fix bug
update code
```

#### 6. TypeScript

**Purpose:** Type safety and compile-time error detection

**Configuration:** `tsconfig.json` with strict mode

**Key Settings:**
- `strict: true` - Enables all strict type checking options
- `noImplicitAny: true` - Prevents implicit any types
- `strictNullChecks: true` - Null and undefined checking
- `noUnusedLocals: true` - Error on unused local variables
- `noUnusedParameters: true` - Error on unused parameters

#### 7. Compodoc

**Purpose:** Automatic API documentation generation

**Usage:**
```bash
# Generate documentation
npm run compodoc:build

# Generate and serve documentation
npm run compodoc:build-and-serve

# Serve existing documentation
npm run compodoc:serve
```

**Features:**
- Extracts JSDoc/TSDoc comments
- Generates interactive documentation
- Includes component, service, and module documentation

## Workflow Integration

### Pre-Commit Workflow

1. Developer stages files: `git add .`
2. Developer commits: `git commit -m "feat: add new feature"`
3. **Husky pre-commit hook triggers:**
   - `lint-staged` runs on staged files
   - Prettier formats files
   - ESLint checks and auto-fixes issues
   - If errors remain, commit is blocked
4. **Husky commit-msg hook triggers:**
   - Commitlint validates message format
   - If invalid, commit is rejected with error message
5. If all checks pass, commit succeeds

### Development Workflow

```bash
# 1. Make code changes
# Edit files...

# 2. Stage changes
git add .

# 3. Commit (hooks run automatically)
git commit -m "feat(component): add new feature"

# If linting errors exist, they're auto-fixed
# If commit message is invalid, you'll see an error
```

### Manual Quality Checks

```bash
# Run all linting checks
npm run lint

# Auto-fix linting issues
npm run lint:fix

# Format all code
npm run prettier

# Run tests
npm run test
```

## Best Practices

### 1. Let Tools Do the Work

```bash
# ✅ Correct - let Prettier format
npm run prettier

# ❌ Incorrect - manual formatting
# Manually adjusting spacing, quotes, etc.
```

### 2. Fix Linting Errors Before Committing

```bash
# ✅ Correct - fix issues first
npm run lint:fix
git add .
git commit -m "fix: resolve linting errors"

# ❌ Incorrect - commit with errors
git commit -m "feat: new feature"  # Will be blocked by hooks
```

### 3. Use Conventional Commits

```bash
# ✅ Correct - follows convention
git commit -m "feat(auth): add password reset"
git commit -m "fix(api): handle null response"

# ❌ Incorrect - doesn't follow convention
git commit -m "added password reset"
git commit -m "fixed bug"
```

### 4. Run Quality Checks Locally

```bash
# ✅ Correct - check before pushing
npm run lint
npm run test
git push

# ❌ Incorrect - rely only on CI
git push  # May fail in CI
```

### 5. Keep Configuration Updated

- Update ESLint rules as Angular evolves
- Keep Prettier configuration consistent
- Review and update commitlint rules as needed

## Consequences

### Positive

- **Consistency:** All code follows the same style automatically
- **Quality:** Errors caught early in development
- **Productivity:** Less time spent on code reviews for style issues
- **History:** Clean, searchable commit history
- **Onboarding:** New developers follow standards automatically
- **Maintainability:** Consistent code is easier to maintain
- **Automation:** Minimal manual intervention required

### Negative

- **Initial Setup:** Requires configuration and setup time
- **Learning Curve:** Team needs to learn tool configurations
- **Commit Rejection:** Invalid commits are rejected (though this is positive)
- **CI/CD Dependency:** Relies on tools working correctly

### Mitigation

- Clear documentation of all tools and configurations
- Team training on tool usage
- Regular updates to tool configurations
- CI/CD pipeline as backup validation

## CI/CD Integration

These tools should also be integrated into CI/CD pipeline:

```yaml
# Example CI/CD steps
- name: Lint code
  run: npm run lint

- name: Format check
  run: npm run prettier -- --check

- name: Run tests
  run: npm run test

- name: Build
  run: npm run build
```

## Updating Tools

When updating tools:

1. **Update dependencies:** `npm update`
2. **Test locally:** Run all quality checks
3. **Update configurations:** Adjust rules as needed
4. **Document changes:** Update this ADR if significant changes
5. **Team communication:** Notify team of breaking changes
