# ADR-001: Separation of Responsibilities - Core, Shared, and Features

**📖 [English](./ADR-001.md) | [Español](../../es/adr/ADR-001.md)**

**Status:** Accepted  
**Date:** 2025
**Deciders:** Architecture Team

## Context

As the application grows, it becomes crucial to maintain a clear separation of concerns to ensure:
- Code reusability across different features
- Independent development and testing of features
- Easy maintenance and refactoring
- Prevention of circular dependencies
- Clear understanding of where code belongs

Without a well-defined structure, developers might place code in inappropriate locations, leading to tight coupling, difficulty in testing, and challenges when scaling the application.

## Decision

We will organize the application into four distinct layers with clear responsibilities:

### Core Layer (`src/app/core/`)

Contains infrastructure and global logic that is used across the entire application:

- **Services:** Global singleton services (e.g., `HttpClient`, `Logger`, `NotificationService`)
- **Guards:** Route guards for authentication and authorization
- **Interceptors:** HTTP interceptors for request/response handling
- **Models:** Global data models and interfaces used across features
- **Enums:** Shared enumerations (e.g., `UserRole`, `HttpStatus`)
- **Constants:** Global configuration and constants
- **Utils:** Helper functions for common operations
- **Pipes:** Common transformation pipes
- **Directives:** Global directives
- **Components:** Global UI components (e.g., main layout, error pages)

**Key principle:** Core should not depend on Shared, Cross, or Features.

### Shared Layer (`src/app/shared/`)

Contains reusable UI components and utilities that can be used by multiple features:

- **Components:** Reusable presentation components (buttons, modals, cards, form fields)
- **Directives:** Shared attribute or structural directives
- **Pipes:** UI formatting pipes (e.g., currency, date formatting)
- **Validators:** Common form validators
- **Types/Interfaces:** Shared type definitions for UI components

**Key principle:** Shared should not depend on Features, but can depend on Core and Cross.

### Cross Layer (`src/app/cross/{domain-area}/`)

Contains domain-level capabilities that cut across multiple features and usually encapsulate business rules, orchestration, or integrations that should not live inside Shared because they are not UI-centric:

- **Services:** Transversal business services (e.g., `AuthService`, `SessionService`)
- **Repositories:** API or storage access shared across domains
- **Managers:** Coordinators for flows that span multiple features (sign-in, analytics bootstrapping)
- **Models:** Domain models reused across features beyond low-level infrastructure
- **Policies/Guards:** Rules that apply in multiple contexts (feature toggles, capability checks)
- **Pipes/Validators:** Domain-specific helpers that do not belong to Shared UI utilities

**Key principle:** Cross depends exclusively on Core.

### Features Layer (`src/app/features/{feature-name}/`)

Contains domain-specific functionality organized by feature:

- **Components:** Feature-specific components
- **Services:** Business logic services for the feature
- **Repositories:** Data access layer for API communication
- **Managers:** State management or business logic orchestration
- **Models:** Feature-specific data models
- **Guards:** Feature-specific route guards
- **Pipes:** Feature-specific transformation pipes
- **Directives:** Feature-specific directives
- **Utils:** Feature-specific helper functions
- **Enums:** Feature-specific enumerations
- **Constants:** Feature-specific constants
- **Routes:** Feature routes defined in `{feature-name}-routes.ts`

**Key principles:**
- Each feature is independent and self-contained
- Features can depend on Core, Shared, and Cross, but not on other Features
- Features should not directly import from other features

## Dependency Rules

```
┌─────────┐
│ Features│
└─┬──┬──┬─┘
  │  │  │ can use
  ▼  ▼  ▼
┌─────────┐    ┌─────────┐
│  Shared │    │  Cross  │
└────┬────┘    └────┬────┘
     │  can use          │ can use
     ▼                   ▼
     └──────────────┌─────────┐
                    │  Core   │
                    └─────────┘
```

### Allowed Dependencies

✅ **Feature → Core:** Allowed
```typescript
// sales/services/sales.service.ts
import { Logger } from '@/core/services/logger.service';
import { User } from '@/core/models/user.model';
```

✅ **Feature → Shared:** Allowed
```typescript
// sales/components/sales-list.component.ts
import { ButtonComponent } from '@/shared/components/button/button.component';
import { CurrencyPipe } from '@/shared/pipes/currency.pipe';
```

✅ **Feature → Cross:** Allowed
```typescript
// sales/services/sales-auth.facade.ts
import { AuthService } from '@/cross/auth/services/auth.service';
```

✅ **Shared → Core:** Allowed
```typescript
// shared/components/modal/modal.component.ts
import { NotificationService } from '@/core/services/notification.service';
```

✅ **Shared → Cross:** Allowed
```typescript
// shared/components/modal/modal.component.ts
import { SessionService } from '@/cross/auth/services/session.service';
```

✅ **Cross → Core:** Allowed
```typescript
// cross/auth/services/auth.service.ts
import { NotificationService } from '@/core/services/notification.service';
```

### Forbidden Dependencies

❌ **Core → Shared:** Not allowed
```typescript
// ❌ DON'T: core/services/logger.service.ts
import { ButtonComponent } from '@/shared/components/button/button.component';
```

❌ **Core → Feature:** Not allowed
```typescript
// ❌ DON'T: core/guards/auth.guard.ts
import { SalesService } from '@/features/sales/services/sales.service';
```

❌ **Core → Cross:** Not allowed
```typescript
// ❌ DON'T: core/interceptors/http-error.interceptor.ts
import { AuthService } from '@/cross/auth/services/auth.service';
```

❌ **Shared → Feature:** Not allowed
```typescript
// ❌ DON'T: shared/components/button/button.component.ts
import { SalesService } from '@/features/sales/services/sales.service';
```

❌ **Cross → Shared or Feature:** Not allowed
```typescript
// ❌ DON'T: cross/auth/services/auth.service.ts
import { ButtonComponent } from '@/shared/components/button/button.component';
// ❌ DON'T: cross/auth/policies/auth-policy.ts
import { SalesService } from '@/features/sales/services/sales.service';
```

❌ **Feature → Feature:** Not allowed
```typescript
// ❌ DON'T: sales/services/sales.service.ts
import { PaymentService } from '@/features/payments/services/payment.service';
```

## Route Definition

Each feature defines its own routes in a `{feature-name}-routes.ts` file located inside `src/app/features/{feature-name}/`:

```typescript
// app/features/sales/sales-routes.ts
import { Routes } from '@angular/router';
import { SalesListComponent } from './components/sales-list/sales-list.component';
import { SaleDetailComponent } from './components/sale-detail/sale-detail.component';

export const salesRoutes: Routes = [
  {
    path: '',
    component: SalesListComponent,
  },
  {
    path: ':id',
    component: SaleDetailComponent,
  },
];
```

Then, these routes are imported in the main app routes:

```typescript
// app.routes.ts
import { salesRoutes } from './features/sales/sales-routes';

export const routes: Routes = [
  {
    path: 'sales',
    loadChildren: () => import('./features/sales/sales-routes').then(m => m.salesRoutes),
  },
  // ... other routes
];
```

## Consequences

### Positive

- **Clear separation of concerns:** Developers know exactly where to place new code
- **Reusability:** Shared components and Core services can be used across features
- **Testability:** Features can be tested independently with mocked Core/Shared dependencies
- **Scalability:** New features can be added without affecting existing ones
- **Maintainability:** Changes in one feature don't impact others
- **Onboarding:** New developers can quickly understand the codebase structure

### Negative

- **Initial setup complexity:** Requires discipline to maintain the structure
- **Potential code duplication:** Similar logic might exist in different features (acceptable if truly feature-specific)
- **Learning curve:** New team members need to understand the dependency rules

### Mitigation

- Document the structure clearly (this ADR)
- Use path aliases (`@/core`, `@/shared`) to make imports clear
- Enforce rules through linting (ESLint rules can prevent forbidden imports)
- Regular code reviews to ensure compliance
