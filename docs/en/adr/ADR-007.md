# ADR-007: Testing Strategy

**📖 [English](./ADR-007.md) | [Español](../../es/adr/ADR-007.md)**

**Status:** Accepted  
**Date:** 2025  
**Deciders:** Architecture Team

## Context

As the application grows, maintaining code quality and preventing regressions becomes critical. Without a well-defined testing strategy:

- Tests may be inconsistent in structure and quality
- Critical bugs might go undetected
- Refactoring becomes risky without test coverage
- Onboarding new developers is slower without test examples
- Integration issues are discovered late in development
- No clear guidelines on what to test and how

We need a comprehensive testing strategy that:
- Ensures code reliability and maintainability
- Provides confidence when refactoring
- Serves as documentation for code behavior
- Catches bugs early in the development cycle
- Maintains consistent test quality across the team

## Decision

We will use a **multi-layered testing approach** with:

1. **Unit Testing:** Vitest for testing individual units (components, services, utilities) in isolation
2. **Integration Testing:** Testing component-service interactions
3. **E2E Testing:** Playwright for end-to-end user workflows
4. **AAA Pattern:** Arrange, Act, Assert structure for all tests
5. **Test Coverage:** Aim for ≥80% branch coverage, focusing on critical paths
6. **Test Isolation:** Tests must be independent and deterministic

## Implementation

### Testing Stack

- **Vitest:** Unit and integration testing framework
- **@analogjs/vite-plugin-angular:** Angular-specific Vitest configuration
- **Playwright:** End-to-end testing framework
- **Testing Utilities:** Angular testing utilities and Material harnesses

### Unit Testing with Vitest

#### Test File Structure

```typescript
describe('ComponentName', () => {
  // Arrange
  let component: ComponentName;
  let fixture: ComponentFixture<ComponentName>;

  beforeEach(() => {
    // Setup
  });

  describe('MethodName', () => {
    it('should do something when condition is met', () => {
      // Arrange
      const input = 'test value';
      
      // Act
      const result = component.methodName(input);
      
      // Assert
      expect(result).toBe('expected value');
    });

    it('should handle error case', () => {
      // Arrange
      const invalidInput = null;
      
      // Act & Assert
      expect(() => component.methodName(invalidInput)).toThrow();
    });
  });
});
```

#### AAA Pattern (Arrange, Act, Assert)

All tests must follow the AAA pattern:

```typescript
it('should calculate total price correctly', () => {
  // Arrange: Set up test data and conditions
  const items = [
    { price: 10, quantity: 2 },
    { price: 5, quantity: 3 }
  ];
  component.items = items;

  // Act: Execute the code under test
  const total = component.calculateTotal();

  // Assert: Verify the expected outcome
  expect(total).toBe(35);
});
```

#### Testing Components

```typescript
import { ComponentFixture, TestBed } from '@angular/core/testing';
import { CustomerListComponent } from './customer-list.component';
import { CustomerRepository } from '@/features/customers/repositories/customer-repository';
import { of } from 'rxjs';

describe('CustomerListComponent', () => {
  let component: CustomerListComponent;
  let fixture: ComponentFixture<CustomerListComponent>;
  let mockRepository: Partial<CustomerRepository>;

  beforeEach(async () => {
    // Arrange: Create mock repository
    mockRepository = {
      findBy: vi.fn().mockReturnValue({
        load: vi.fn().mockResolvedValue([
          { id: '1', name: 'John' },
          { id: '2', name: 'Jane' }
        ]),
        value: signal([]),
        loading: signal(false),
        error: signal(null),
        destroy: vi.fn()
      })
    } as any;

    await TestBed.configureTestingModule({
      imports: [CustomerListComponent],
      providers: [
        { provide: CustomerRepository, useValue: mockRepository }
      ]
    }).compileComponents();

    fixture = TestBed.createComponent(CustomerListComponent);
    component = fixture.componentInstance;
  });

  it('should load customers on init', async () => {
    // Arrange
    const loadSpy = vi.spyOn(mockRepository.findBy()!, 'load');

    // Act
    component.ngOnInit();
    await fixture.whenStable();

    // Assert
    expect(loadSpy).toHaveBeenCalled();
  });

  it('should display customers when loaded', () => {
    // Arrange
    component.customers = {
      value: signal([{ id: '1', name: 'John' }]),
      loading: signal(false),
      error: signal(null),
      load: vi.fn(),
      destroy: vi.fn()
    } as any;
    fixture.detectChanges();

    // Act
    const customerElements = fixture.nativeElement.querySelectorAll('.customer-item');

    // Assert
    expect(customerElements.length).toBe(1);
  });
});
```

#### Testing Services

```typescript
import { TestBed } from '@angular/core/testing';
import { HttpClientTestingModule, HttpTestingController } from '@angular/common/http/testing';
import { CustomerRepository } from './customer-repository';

describe('CustomerRepository', () => {
  let repository: CustomerRepository;
  let httpMock: HttpTestingController;

  beforeEach(() => {
    TestBed.configureTestingModule({
      imports: [HttpClientTestingModule],
      providers: [CustomerRepository]
    });

    repository = TestBed.inject(CustomerRepository);
    httpMock = TestBed.inject(HttpTestingController);
  });

  afterEach(() => {
    httpMock.verify();
  });

  it('should fetch customer by id', async () => {
    // Arrange
    const customerId = '123';
    const mockCustomer = { id: customerId, name: 'John' };

    // Act
    const resource = repository.find();
    const promise = resource.load(customerId);

    // Assert
    const req = httpMock.expectOne(`/api/v1/customers/${customerId}`);
    expect(req.request.method).toBe('GET');
    req.flush(mockCustomer);

    const result = await promise;
    expect(result).toEqual(mockCustomer);
    expect(resource.value()).toEqual(mockCustomer);
  });
});
```

#### Testing Signals

```typescript
describe('CustomerService', () => {
  let service: CustomerService;

  beforeEach(() => {
    service = new CustomerService();
  });

  it('should initialize with empty customers', () => {
    // Arrange & Act
    const customers = service.customers();

    // Assert
    expect(customers).toEqual([]);
  });

  it('should update customers signal', () => {
    // Arrange
    const newCustomers = [{ id: '1', name: 'John' }];

    // Act
    service.setCustomers(newCustomers);

    // Assert
    expect(service.customers()).toEqual(newCustomers);
  });

  it('should compute total count', () => {
    // Arrange
    service.setCustomers([
      { id: '1', name: 'John' },
      { id: '2', name: 'Jane' }
    ]);

    // Act
    const count = service.totalCount();

    // Assert
    expect(count).toBe(2);
  });
});
```

### Testing Best Practices

#### 1. Descriptive Test Names

```typescript
// ✅ Correct - describes expected behavior
it('should display error message when API call fails', () => {});
it('should disable submit button when form is invalid', () => {});
it('should filter customers by search term', () => {});

// ❌ Incorrect - vague or implementation-focused
it('should work', () => {});
it('should call method', () => {});
it('test1', () => {});
```

#### 2. Test Both Positive and Negative Cases

```typescript
describe('validateEmail', () => {
  it('should return true for valid email', () => {
    expect(validateEmail('test@example.com')).toBe(true);
  });

  it('should return false for invalid email', () => {
    expect(validateEmail('invalid-email')).toBe(false);
  });

  it('should return false for empty string', () => {
    expect(validateEmail('')).toBe(false);
  });

  it('should return false for null', () => {
    expect(validateEmail(null as any)).toBe(false);
  });
});
```

#### 3. Use Test Doubles (Mocks, Spies, Stubs)

```typescript
import { vi } from 'vitest';

// Mock dependencies
const mockRepository = {
  findBy: vi.fn().mockReturnValue({
    load: vi.fn(),
    value: signal([]),
    loading: signal(false),
    error: signal(null),
    destroy: vi.fn()
  })
} as any;

// Spy on methods
const loadSpy = vi.spyOn(component, 'loadData');

// Stub external services
vi.mock('app/core/services/logger', () => ({
  Logger: {
    log: vi.fn()
  }
}));
```

#### 4. Test Public APIs, Not Implementation Details

```typescript
// ✅ Correct - test public behavior
it('should update customer name', () => {
  component.updateCustomerName('123', 'New Name');
  expect(component.getCustomer('123').name).toBe('New Name');
});

// ❌ Incorrect - test implementation details
it('should call internal method', () => {
  const spy = jest.spyOn(component as any, 'internalMethod');
  component.updateCustomerName('123', 'New Name');
  expect(spy).toHaveBeenCalled();
});
```

#### 5. Keep Tests Isolated

```typescript
// ✅ Correct - each test is independent
describe('Calculator', () => {
  let calculator: Calculator;

  beforeEach(() => {
    calculator = new Calculator(); // Fresh instance for each test
  });

  it('should add numbers', () => {
    expect(calculator.add(2, 3)).toBe(5);
  });

  it('should subtract numbers', () => {
    expect(calculator.subtract(5, 3)).toBe(2);
  });
});

// ❌ Incorrect - tests depend on each other
let calculator = new Calculator(); // Shared state

it('should add numbers', () => {
  calculator.add(2, 3);
  expect(calculator.result).toBe(5);
});

it('should subtract from previous result', () => {
  calculator.subtract(2); // Depends on previous test
  expect(calculator.result).toBe(3);
});
```

#### 6. Use Angular Material Harnesses

```typescript
import { MatButtonHarness } from '@angular/material/button/testing';
import { TestbedHarnessEnvironment } from '@angular/cdk/testing/testbed';

it('should disable button when form is invalid', async () => {
  // Arrange
  component.form.setErrors({ invalid: true });
  fixture.detectChanges();

  // Act
  const loader = TestbedHarnessEnvironment.loader(fixture);
  const button = await loader.getHarness(MatButtonHarness);

  // Assert
  expect(await button.isDisabled()).toBe(true);
});
```

### Test Coverage Goals

- **Branch Coverage:** ≥80% for critical paths
- **Function Coverage:** ≥80% overall
- **Line Coverage:** ≥80% overall
- **Focus Areas:** Business logic, error handling, edge cases

### E2E Testing with Playwright

For end-to-end testing, use Playwright to test complete user workflows:

```typescript
import { test, expect } from '@playwright/test';

test('user can login and view dashboard', async ({ page }) => {
  // Arrange: Navigate to login page
  await page.goto('/signin');

  // Act: Fill login form and submit
  await page.fill('[formControlName="email"]', 'user@example.com');
  await page.fill('[formControlName="password"]', 'password123');
  await page.click('button[type="submit"]');

  // Assert: Verify redirect to dashboard
  await expect(page).toHaveURL('/dashboard');
  await expect(page.locator('h1')).toContainText('Dashboard');
});
```

## Test Organization

### File Naming

- Test files: `*.spec.ts` (e.g., `customer.service.spec.ts`)
- E2E tests: `*.e2e.spec.ts` (e.g., `login.e2e.spec.ts`)

```
src/
├── app/
│   ├── components/
│   │   └── customer-list/

│   │       └── customer-list.component.spec.ts

│       └── customer.service.spec.ts
└── e2e/
    └── login.e2e.spec.ts
```

## Consequences

### Positive

- **Code Quality:** Tests catch bugs early and prevent regressions
- **Confidence:** Safe refactoring with test coverage
- **Documentation:** Tests serve as executable documentation
- **Maintainability:** Tests help understand code behavior
- **Team Collaboration:** Consistent test patterns across team
- **CI/CD Integration:** Automated testing in deployment pipeline

### Negative

- **Time Investment:** Writing tests requires additional development time
- **Maintenance:** Tests need to be updated when code changes
- **Learning Curve:** Team members need to learn testing patterns
- **Test Brittleness:** Tests can break due to implementation changes

### Mitigation

- Start with critical paths and high-risk areas
- Use test generators and templates
- Regular test reviews to ensure quality
- Balance between unit and integration tests
- Focus on testing behavior, not implementation

## Running Tests

```bash
# Run tests in watch mode

# Run tests with coverage
npm run test -- --coverage

# Run E2E tests

```
