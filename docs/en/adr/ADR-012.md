# ADR-012: Access modifiers and `readonly` convention in TypeScript

**📖 [English](./ADR-012.md) | [Español](../../es/adr/ADR-012.md)**

**Status:** Accepted  
**Date:** 2025-12-04  
**Deciders:** Architecture Team

## Context

The project is written primarily in TypeScript and uses Angular. TypeScript allows omitting access modifiers (`public`, `private`, `protected`), which can cause inconsistencies about which class members are part of the public API and which are internal implementation details. In addition, omitting `readonly` for properties that should not change can lead to unexpected state mutations.

We need a clear convention to improve consistency, avoid accidental mutations, and remain compatible with Angular conventions.

## Decision

We adopt the following rules across the codebase:

1. **Always specify access modifiers explicitly**

- `public`: members that form part of the class public API.  
- `private`: implementation details.  
- `protected`: members intended for subclasses (inheritance).

2. **Use `readonly` whenever a property should not change after initialization**

This applies to IDs, configuration values, injected service references that must not mutate, and other instance constants.

3. **Constructor parameters**: use access modifiers only when the parameter becomes a class property.

Correct example:

```typescript
constructor(
  public name: string,
  private readonly service: ApiService,
  readonly id: number
) {}
```

If a constructor parameter is not a property, do not add modifiers in the constructor signature.

4. **Exception: Angular lifecycle hooks**

Angular lifecycle hook methods (`ngOnInit`, `ngAfterViewInit`, `ngOnDestroy`, `ngOnChanges`, etc.) SHOULD NOT include an explicit `public` modifier. Write them like:

```typescript
ngOnInit(): void {
  // ...
}
```

Rationale: Angular style guidance does not use `public` on hooks and hooks are public by convention and detected by name.

5. **Linting rules**

Configure ESLint to warn when:
- an access modifier is missing on a developer-defined property or method, and
- a property that should be `readonly` is mutated.

Exclude Angular lifecycle hooks from the rule that enforces an explicit `public` modifier.

## Implementation

- Update ESLint/TSLint rules to emit warnings for missing modifiers and for unexpected mutations of `readonly` properties.  
- Create a gradual refactor task to bring existing classes into compliance (prioritize new files and PR changes).  
- Add examples and a guideline entry in the repository describing the rules and usage examples.

## Consequences

### Positive

- Greater clarity and consistency in the codebase.  
- Fewer bugs caused by unexpected mutations.  
- Easier to identify a class public API.  
- Better alignment with TypeScript best practices.

### Negative

- More typing when declaring class members.  
- Existing code will require adaptation to follow the convention.

### Mitigation

- Adopt the convention gradually (new code and PRs first).  
- Use lint warnings before enforcing errors.  
- Provide clear repository guidelines and examples.
