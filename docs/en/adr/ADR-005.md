# ADR-005: Internationalization (i18n) Strategy

**📖 [English](./ADR-005.md) | [Español](../../es/adr/ADR-005.md)**

**Status:** Accepted  
**Date:** 2025  
**Deciders:** Architecture Team

## Context

The application needs to support multiple languages to serve a global audience. Key requirements include:

- Support for multiple languages without rebuilding the application
- Runtime language switching
- Maintainable translation workflow
- Synchronization between source code and translation files
- Detection of missing translations
- English as the base language for all translations
- Integration with Angular's built-in i18n capabilities

Without a well-defined i18n strategy, we risk:
- Hardcoded strings in the UI
- Inconsistent translation management
- Difficult maintenance of translation files
- Poor developer experience when adding new strings
- Missing translations going unnoticed

## Decision

We will use **Angular's built-in i18n** with **runtime translation loading** using `loadTranslations()` from `@angular/localize`. The implementation includes:

1. **Angular i18n attributes** in templates for string extraction
2. **JSON as base format** (`en.json`) extracted from source code
3. **JavaScript modules** (`.js` files) for runtime loading
4. **Custom script** (`generate-i18n.js`) for translation file management
5. **Runtime locale detection** and dynamic translation loading
6. **Missing translation detection** through automated tooling

## Implementation

### File Structure

```
public/
└── i18n/
    ├── en.json          # Base English translations (extracted by Angular)
    ├── en.js            # Generated JavaScript module for English
    ├── es.js            # Spanish translations
    ├── fr.js            # French translations
    ├── es_missing.json  # Missing Spanish translations (generated)
    └── ...
```

### Translation Workflow

#### 1. Adding New Strings

Developers add the `i18n` attribute to translatable strings in templates:

```html
<!-- ✅ Correct - Using i18n attribute -->
<button mat-flat-button color="primary" i18n>Save</button>
<mat-label i18n>Username or email</mat-label>
<div i18n>Welcome, {{ userName() }}</div>

<!-- ❌ Incorrect - Hardcoded strings -->
<button mat-flat-button color="primary">Save</button>
<mat-label>Username or email</mat-label>
```

#### 2. Extracting Translations

Run the extraction command to generate `en.json`:

```bash
npm run extract-i18n
```

This command:
- Scans all templates for `i18n` attributes
- Extracts strings and generates unique IDs
- Creates `public/i18n/en.json` with all English strings

#### 3. Generating Language Files

To generate or update a language file:

```bash
npm run i18n -- es    # Generate/update Spanish
npm run i18n -- fr    # Generate/update French
```

The `generate-i18n.js` script:
- Reads `en.json` (base translations)
- Compares with existing target language file (e.g., `es.js`)
- Generates/updates the target language JS file
- Creates `{lang}_missing.json` for missing translations
- Generates `en.js` from `en.json` for runtime loading

#### 4. Translation File Format

**Base JSON format** (`en.json`):
```json
{
  "translations": {
    "439130533561834924": " Accept ",
    "1519954996184640001": "Error",
    "5207635742003539443": "Sign in",
    "8164791295086777340": "Don't have an account?",
    ...
  }
}
```

**Generated JS format** (`es.js`):
```javascript
export default {
  '439130533561834924': ' Aceptar ',
  '1519954996184640001': 'Error',
  '5207635742003539443': 'Iniciar sesión',
  '8164791295086777340': '¿No tienes una cuenta?',
  ...
};
```

**Missing translations** (`es_missing.json`):
```json
{
  "1234567890123456789": "New string that needs translation"
}
```

### Runtime Translation Loading

The `AppManager` service handles locale detection and translation loading:

```typescript
private async setLocale(): Promise<string> {
  // 1. Detect locale from user preference, system, or default
  const systemLocale = isPlatformBrowser(this.platformId)
    ? this.languages().find((l) => l.code === navigator.language.split('-')[0])?.code
    : null;
  const userLocale = this.languages().find(
    (l) => l.code === this.storageService.get(this.localeKey, 'local')
  )?.code;
  const locale = userLocale || systemLocale || this.defaultLocale;
  
  // 2. Store user preference
  this.storageService.set(this.localeKey, locale, 'local');
  
  // 3. Dynamically import translation file
  const localeTranslationsModule = await import(`../../../../public/i18n/${locale}.js`);
  
  // 4. Load translations at runtime
  loadTranslations(localeTranslationsModule.default);
  
  // 5. Configure moment.js for date/time localization
  moment.locale(locale);
  
  return locale;
}
```

### Locale Detection Priority

1. **User preference** (stored in localStorage)
2. **System locale** (browser's `navigator.language`)
3. **Default locale** (English - `en`)

### Language Switching

Users can switch languages through the settings component, which updates the stored preference and reloads translations:

```typescript
changeLanguage(language: Language): void {
  this.appManager.setLocale(language.code);
  // AppManager handles translation reloading
}
```

## Key Features

### 1. Runtime Loading

- Translations are loaded dynamically at runtime
- No need to rebuild the application for new languages
- Supports lazy loading of translation files

### 2. Missing Translation Detection

The `generate-i18n.js` script automatically:
- Identifies missing translations
- Creates `{lang}_missing.json` files
- Removes missing files when all translations are complete

### 3. Translation Synchronization

- Base file (`en.json`) is the source of truth
- All language files are synchronized with the base
- Keys are automatically ordered according to the base file

### 4. Security

The `generate-i18n.js` script includes security measures:
- Language code validation (alphanumeric only)
- Path traversal prevention
- Safe file resolution

### Example 5: Using $localize in TypeScript Classes

When you need to translate strings from TypeScript code (services, components, etc.), use the `$localize` function:

```typescript
import { Component, inject } from '@angular/core';
import { NotificationService } from '@factor_ec/utils';

@Component({
  selector: 'ft-user-profile',
  standalone: true,
  template: `
    <div>
      <h1>{{ title }}</h1>
      <button (click)="saveUser()">{{ saveButtonText }}</button>
    </div>
  `
})
export class UserProfileComponent {
  private notificationService = inject(NotificationService);
  
  // ✅ Correct - Using $localize for translatable strings
  title = $localize`User Profile`;
  saveButtonText = $localize`Save Changes`;
  
  saveUser(): void {
    // ✅ Correct - Using $localize with interpolation
    const userName = 'John Doe';
    this.notificationService.success(
      $localize`User ${userName} has been saved successfully.`
    );
  }
  
  showError(): void {
    // ✅ Correct - Using $localize for error messages
    this.notificationService.error(
      $localize`An error occurred while saving the user. Please try again.`
    );
  }
}
```

**Important Notes:**
- `$localize` uses template literal syntax with backticks (`` ` ``)
- Interpolation is done using `${variable}` syntax
- The string will be extracted during the `npm run extract-i18n` process
- The extracted string will appear in `en.json` with a generated ID

**Generated in en.json:**
```json
{
  "translations": {
    "1234567890123456789": "User Profile",
    "9876543210987654321": "Save Changes",
    "5555555555555555555": "User {$PH} has been saved successfully.",
    "1111111111111111111": "An error occurred while saving the user. Please try again."
  }
}
```

## Development Workflow

### Adding a New Language

1. **Extract base translations:**
   ```bash
   npm run extract-i18n
   ```

2. **Generate language file:**
   ```bash
   npm run i18n -- de  # For German
   ```

3. **Translate missing strings:**
   - Open `public/i18n/de_missing.json`
   - Translate all strings
   - Copy translations to `public/i18n/de.js`

4. **Regenerate to verify:**
   ```bash
   npm run i18n -- de
   ```
   - If all translations are complete, `de_missing.json` will be removed

### Updating Translations

1. **Add new strings** to templates with `i18n` attribute
2. **Extract** new strings: `npm run extract-i18n`
3. **Update language files:** `npm run i18n -- es`
4. **Translate** new strings in `{lang}_missing.json`
5. **Regenerate** to apply translations

## Rules and Guidelines

### Code Rules

- **All UI-visible text must use `i18n` attribute** in templates
- **For TypeScript classes**, use `$localize` for translatable strings
- **No hardcoded strings** in templates or TypeScript code
- **Code comments and documentation** must be in English
- **Variable and function names** must be in English

### Translation Guidelines

- **Base language:** Always English (`en`)
- **Translation quality:** Use professional translators when possible
- **Context:** Provide context comments for translators when needed
- **Consistency:** Use consistent terminology across translations
- **Pluralization:** Angular handles pluralization automatically

## Consequences

### Positive

- **Runtime Language Switching:** Users can change language without page reload
- **No Rebuild Required:** New languages can be added without rebuilding
- **Maintainable Workflow:** Automated tools simplify translation management
- **Missing Translation Detection:** Automated detection of untranslated strings
- **Type Safety:** Translation IDs are generated and tracked
- **Angular Integration:** Uses Angular's built-in i18n capabilities
- **Developer Experience:** Simple `i18n` attribute for marking translatable strings

### Negative

- **Large Translation Files:** All translations loaded at once (can be optimized with lazy loading per feature)
- **Numeric IDs:** Angular generates numeric IDs which are not human-readable
- **Build Step Required:** Must run extraction before generating language files
- **Manual Translation:** Requires manual translation work (though tooling helps)
- **File Management:** Multiple files to manage (JSON, JS, missing files)

### Mitigation

- **Lazy Loading:** Consider feature-based translation loading for large applications
- **Translation Management:** Use translation management tools for better workflow
- **Pluralization Rules:** Ensure proper pluralization for all languages
- **Automated Testing:** Add tests to detect missing translations
- **Documentation:** Clear documentation for translators and developers
- **CI/CD Integration:** Automate translation extraction in CI/CD pipeline
